# Day18：AI Agent架构——从ReAct到自主决策（番茄86-90）

> **Part 4 · 大语言模型与智能体**
> **悬疑探案进度**：AI从"被动回答"进化到"主动行动"——这是智能体革命的起点。
> **番茄范围**：🍅86-90

---

## 悬疑开场：当AI不再只是"回答问题"

2023年，人们用AI做这件事：

```
用户：帮我写一封邮件。
AI：好的，这是邮件草稿。
用户：再改一下。
AI：改好了。
```

**每一步都要人下指令。** 你是指挥官，AI是执行者。

2026年，人们用AI做这件事：

```
用户：帮我预订今晚7点两个人在老地方的晚餐座位。
AI：让我来处理…
[1] 检查我日历 → 今晚空着
[2] 搜索"老地方" → 发现是"老地方火锅"
[3] 查询餐厅官网 → 支持在线预订
[4] 打开预订页面 → 选"今晚19:00，2位"
[5] 填写信息 → 用我的默认联系方式
[6] 确认预订 → 成功！
[7] 把预订信息写入日历
[8] 发一条确认消息给我

AI：已预订成功！老地方火锅，今晚19:00，2位。信息已加入日历。
```

**AI自己规划、自己执行、自己在出错时调整。** 你只说了目标，AI完成了全过程。

这就是从"LLM聊天"到"AI Agent"的跨越。

但Agent到底是什么？它怎么做到的？为什么不是所有LLM都能当Agent？

这就是今天要破的案。

## 🍅 番茄86 · T1：Agent是什么——LLM+工具+记忆+规划

### 🔍 犯罪现场：一个AI Agent的"解剖"

**Agent ≠ LLM。**

Agent是一个**系统**——LLM是它的"大脑"，但还需要其他组件才能"活"起来。

### 🧠 四要素框架

```
                    ┌─────────────────────────┐
                    │      AI Agent           │
                    │                         │
                    │  ┌───────────────────┐  │
                    │  │   🧠 LLM（大脑）    │  │  ← 思考、推理、决策
                    │  └───────────────────┘  │
                    │           │             │
                    │  ┌────────┴────────┐   │
                    │  │                 │   │
                    │  ▼                 ▼   │
                    │ 工具           记忆     │  ← 行动、记住
                    │ (Function Call) (Memory)│
                    │                         │
                    │  ┌───────────────────┐  │
                    │  │   规划（Planning）  │  │  ← 拆解任务、排序
                    │  └───────────────────┘  │
                    └─────────────────────────┘
```

### 🧠 四要素详解

| 要素 | 作用 | 类比人类 | 例子 |
|:-----|:-----|:---------|:-----|
| **LLM（大脑）** | 理解、推理、决策 | 大脑皮层 | GPT-4、Claude、DeepSeek |
| **工具（手脚）** | 执行行动，接触外部世界 | 手、嘴、眼睛 | 搜索、计算器、API调用 |
| **记忆（记忆）** | 保留经验，跨步骤不遗忘 | 海马体 | Context窗口、外部存储 |
| **规划（前额叶）** | 拆解复杂任务，按序执行 | 前额叶皮质 | Chain-of-Thought、任务分解 |

**关键：这四个要素缺一不可。** 只有LLM没有工具？——它只能回答，不能行动。只有LLM没有记忆？——它做了上一步，忘了下一步。

> ✋ **费曼自测**：用"开一家餐厅"做类比——LLM、工具、记忆、规划分别对应餐厅里的什么角色或功能？

### 🧠 Agent的核心能力：从"被动"到"主动"

| 维度 | LLM（聊天） | Agent |
|:-----|:-----------|:------|
| 触发方式 | 用户问→AI答 | 接收目标→自主规划→执行 |
| 任务范围 | 单轮/多轮对话 | 多步任务，可能跨小时/天 |
| 错误处理 | 用户发现→用户修正 | 自我监测→自我修正 |
| 工具使用 | 无（或需用户告知） | 自动判断何时用什么工具 |
| 持久性 | 每次会话独立 | 跨会话记忆，持续学习 |

## 🍅 番茄87 · T2：ReAct模式——Reason+Act：边思考边行动

### 🔍 犯罪现场：为什么"想一步做一步"是最强大的模式？

2022年，研究人员在Google Brain发现了一个简单却强大的模式：

> **让LLM交替"推理"和"行动"，每次行动后观察结果再继续推理——效果远超先全部想好再行动。**

这就是ReAct（Reason + Act）。

### 🧠 ReAct循环

```
循环开始
  ↓
思考（Thought）：我需要知道北京的天气才能回答用户问题
  ↓
行动（Action）：search("北京天气 2026-06-15")
  ↓
观察（Observation）："晴，22-32°C"
  ↓
思考（Thought）：现在知道了天气，我可以回答用户了。另外用户问要不要带伞，不需要。
  ↓
行动（Action）：answer("北京今天晴，22-32°C，不需要带伞。")
  ↓
循环结束（或继续下一轮）
```

**每次循环 = 思考→行动→观察→再思考→再行动……**

### 🧠 ReAct vs 传统方式

```
❌ 传统LLM（无ReAct）：
  用户：今天北京天气怎么样？我该穿什么？
  AI：北京今天……嗯……（没有搜索能力，只能靠训练数据）
      "北京天气多变，建议带伞"（可能过时，可能胡说）

✅ ReAct Agent：
  用户：今天北京天气怎么样？我该穿什么？
  Agent：
  Thought: 我需要查实时天气。
  Action: search_weather("北京")
  Observation: 晴，22-32°C，湿度40%
  Thought: 22-32°C，温差大。早上凉，中午热。建议分层穿搭。
  Action: answer("北京今天晴，22-32°C。早晨偏凉（22°C）建议穿薄外套，中午热（32°C）可穿短袖。不需要带伞。")
```

### 🧠 ReAct的三种变体

| 变体 | 模式 | 适用场景 |
|:-----|:-----|:---------|
| **Standard ReAct** | 思考→行动→观察→循环 | 通用多步任务 |
| **Plan + ReAct** | 先列计划 → 按计划ReAct | 复杂任务（先规划再执行） |
| **Reflexion** | ReAct + 反思（从错误中学习）| 高错误率任务（代码生成） |

**Reflexion的特别之处**：

```
Round 1：
  Thought: 我要写一个函数…
  Action: write_code(...)
  Observation: 运行报错：TypeError
  Reflection: 我忘记处理None的情况了，下次注意

Round 2：
  Thought: 基于上次的反思，我需要加None检查
  Action: fix_code(...)
  Observation: 运行成功
```

### 🧠 ReAct的Prompt结构（简化版）

```
你是一个AI助手，可以调用工具。对于每个步骤：
1. 先思考（Thought）：分析当前状态，决定下一步
2. 再行动（Action）：调用工具
3. 观察结果（Observation）
4. 重复直到任务完成

可用工具：
- search(query): 搜索网络
- calculate(expr): 计算数学表达式
- write_file(path, content): 写文件

---

用户：帮我查一下"苹果公司"的股价，然后计算如果买100股需要多少钱。

Thought: 用户需要实时股价信息，我可以用搜索工具。
Action: search("苹果公司股价 2026-06-15")
...

（Agent继续循环直到完成）
```

### 🧠 Agent的决策循环可视化

```
                    用户给出目标
                        │
                        ▼
                ┌────────────────┐
                │  理解目标       │
                │  拆解为子任务   │
                └───────┬────────┘
                        │
                ┌───────▼────────┐
         ┌──────│  思考下一步     │
         │      │  Thought       │
         │      └───────┬────────┘
         │              │
         │      ┌───────▼────────┐
         │      │  选择工具       │
         │      │  Action        │
         │      └───────┬────────┘
         │              │
         │      ┌───────▼────────┐
         │      │  执行并观察     │
         │      │  Observation   │
         │      └───────┬────────┘
         │              │
         │     ┌────────▼────────┐
         │     │  任务完成？      │
         └─────│  NO → 继续      │
               │  YES → 输出结果  │
               └─────────────────┘
```

## 🍅 番茄88 · T3：Agent记忆系统——短期记忆+长期记忆

### 🔍 犯罪现场：Agent最容易被忽略的组件

大部分失败的Agent项目，失败原因不是"AI不够聪明"，而是**"AI记不住"**。

如果没有记忆系统，Agent就像一个失忆症侦探——每次回到案发现场都要重新开始调查。

### 🧠 两种记忆

| 类型 | 类比人类 | Agent中的体现 | 限制 |
|:-----|:---------|:-------------|:-----|
| **短期记忆** | 你正在思考的事情 | Context Window（当前会话） | 容量有限（4K-200K tokens） |
| **长期记忆** | 你的知识和经验 | 外部存储（文件、数据库） | 需要主动读写 |

### 🧠 短期记忆（Working Memory）

**本质**：当前Context Window里的所有内容。

```
Agent的"短期记忆"包括：
  ┌─ 系统指令（你是谁、你能做什么）
  ├─ 用户目标（"帮我预订餐厅"）
  ├─ 之前的思考过程（Thought历史）
  ├─ 之前的行动结果（Observation历史）
  ├─ 当前中间状态
  └─ 待办子任务列表
```

**短期记忆的挑战**：

```
Context溢出：
  如果Agent执行了20步，每步产生500字的Thought+Observation+Action
  → 20步 × 500字 = 10,000字
  → 继续下去：Context满了，最早的步骤被"挤出去"
  → 结果：Agent开始忘记自己一开始做了什么

解决方案：
  1. 总结历史（Summarize）：定期把旧步骤压缩成摘要
  2. 遗忘机制（Forget）：丢弃不重要的中间步骤
  3. 滑动窗口（Sliding Window）：只保留最近N步
```

### 🧠 长期记忆（External Memory）

**本质**：存储在Agent外部、可以跨会话访问的信息。

```
存储方案：
  
  📝 文件系统（最简单）
  └── agent-memory/
      ├── session-001.md  ← "上一次会话做了什么"
      ├── user-preferences.md  ← "用户的偏好"
      └── knowledge-base/  ← "学到的知识"
  
  🗄️ 数据库（结构化）
  └── Memory Table
      ├── id | type | content | timestamp
      ├── 1  | fact | "用户喜欢火锅" | 2026-06-15
      └── 2  | skill | "学会了Python异步" | 2026-06-14
  
  🧠 向量数据库（语义检索）
  └── 用Embedding存记忆
      查询："和用户饮食偏好相关的记忆"
      结果："用户喜欢火锅"、"用户不吃辣"
```

### 🧠 记忆系统在Agent中的实际工作流

```
用户：帮我安排下周的聚餐，上次去的那家火锅店不错。

Agent的记忆流程：
  1. 读取长期记忆 → 搜索"聚餐"、"火锅"、"偏好"
     → 找到："用户喜欢老地方火锅"
     → 找到："上次聚餐是5月，6个人"
  
  2. 结合短期记忆（用户当前目标和Context）
     → 知道要预订"老地方火锅"
  
  3. 执行预订（调用工具）
     → 预订成功
  
  4. 写入长期记忆
     → "2026-06-22 聚餐预订成功，老地方火锅，4位"
```

> ✋ **费曼自测**：短期记忆和长期记忆在Agent中各自的角色是什么？为什么不能只用一种？

### 🧠 记忆系统的设计原则

| 原则 | 说明 | 反例 |
|:-----|:-----|:-----|
| ① 持久化 | 记忆必须存在Agent外部 | Agent重启后所有记忆丢失 |
| ② 可检索 | 需要时能快速找到相关记忆 | 所有记忆存一个文件，每次全扫一遍 |
| ③ 可遗忘 | 不重要/过时的记忆应能丢弃 | 永不清除，被无效信息淹没 |
| ④ 结构化 | 不同类型记忆分开存储 | 事实和对话历史混在一起 |
| ⑤ 安全 | 敏感信息应加密/权限控制 | Agent把用户密码写进了长期记忆 |

## 🍅 番茄89 · T4：工具使用——Function Calling：让LLM调用API、执行代码、查询数据库

### 🔍 犯罪现场：Agent的"手"是怎么长的？

一个只会"说话"的Agent能做的事情极其有限。

**工具 = Agent连接真实世界的接口。**

### 🧠 Function Calling

Function Calling = LLM的能力边界扩展。它不是让LLM"学会"做某事，而是让LLM**知道"什么时候调用什么工具"**。

```
Agent看到用户说："帮我查一下数据库里昨天的订单数。"

Agent的内部推理：
  1. Thought: 用户需要数据库查询，我有一个query_database工具
  2. Action: query_database("SELECT COUNT(*) FROM orders WHERE date = '2026-06-14'")
  3. Observation: 1427
  4. Answer: "昨天共有1,427个订单。"
```

### 🧠 工具定义的方式（以OpenAI/Claude Function Calling格式为例）

```json
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "query_database",
        "description": "执行SQL查询，返回结果",
        "parameters": {
          "type": "object",
          "properties": {
            "query": {
              "type": "string",
              "description": "SQL查询语句"
            }
          },
          "required": ["query"]
        }
      }
    },
    {
      "type": "function",
      "function": {
        "name": "send_email",
        "description": "发送邮件给指定收件人",
        "parameters": {
          "type": "object",
          "properties": {
            "to": {
              "type": "string",
              "description": "收件人邮箱"
            },
            "subject": {
              "type": "string",
              "description": "邮件主题"
            },
            "body": {
              "type": "string",
              "description": "邮件正文"
            }
          },
          "required": ["to", "subject", "body"]
        }
      }
    }
  ]
}
```

### 🧠 常见Agent工具分类

| 类别 | 工具示例 | 作用 |
|:-----|:---------|:-----|
| 🔍 **搜索** | web_search, wikipedia, news | 获取外部信息 |
| 📝 **文件** | read_file, write_file, edit_file | 操作本地文件 |
| 💻 **代码** | run_python, execute_bash | 执行代码，计算 |
| 🗄️ **数据库** | query_sql, mongodb_find | 数据存取 |
| 🌐 **API** | http_request, call_api | 调用外部服务 |
| ✉️ **通信** | send_email, send_slack | 与人沟通 |
| 🧮 **计算** | calculate, calculator | 精确数学计算 |
| 📅 **日历** | check_calendar, create_event | 时间管理 |

### 🧠 MCP协议——工具的标准化

MCP（Model Context Protocol）是Anthropic推出的工具标准化协议：

```
传统方式：
  每个Agent → 各自定义工具格式 → 不兼容
  换一个Agent框架 → 重新写所有工具定义

MCP方式：
  一次定义工具 → 任何支持MCP的Agent都能用
  工具服务器 ↔ MCP协议 ↔ Agent客户端

就像USB接口：
  没有USB → 每个设备需要专用接口
  有了USB → 统一接口，即插即用
```

> ✋ **费曼自测**：为什么说"工具定义的质量直接影响Agent的可靠性"？用"给一个人一套不靠谱的工具"做类比。

### 🧠 Function Calling的常见陷阱

| 陷阱 | 表现 | 解决方案 |
|:-----|:-----|:---------|
| 过度调用 | Agent没事也调工具（浪费钱） | 工具描述写"只在必要时调用" |
| 参数幻觉 | Agent编造参数（"query_database(table=不存在的表)"） | 参数校验 + 给有限的枚举值 |
| 循环调用 | 反复调同一个工具，结果不变 | 设置最大调用次数 |
| 忽视结果 | 调了工具但不用结果 | Prompt明确要求"基于观察回答" |

## 🍅 番茄90 · T5：多Agent协作——Maker-Checker、Agent团队架构

### 🔍 犯罪现场：一个Agent不够怎么办？

有时候，一个Agent做不了所有事。

- 它既要写代码，又要审查代码——**但它总是觉得自己写得挺好**。
- 它既要创意，又要严谨——**创意和严谨很难在一个"性格"里共存**。
- 它要处理10个并行子任务——**一个Agent串行处理太慢了**。

**解决方案：多Agent系统。**

### 🧠 多Agent的核心思想：专业化+制衡

```
不要造一个"全能型通才Agent"——
造一群"专精型专家Agent"，再给它们一个"管理Agent"。
```

### 🧠 经典模式一：Maker-Checker（编写-审查分离）

这是最基础、最重要的多Agent模式。

```
┌─────────────────────────────────────┐
│            Supervisor Agent          │
│         （分配任务，汇总结果）         │
└──────────┬──────────────┬───────────┘
           │              │
    ┌──────▼──────┐  ┌───▼────────┐
    │ Maker Agent │  │ Checker    │
    │ 写代码的    │  │ Agent      │
    │ 关注速度    │  │ 审代码的    │
    └──────┬──────┘  │ 关注质量    │
           │         └───┬────────┘
           │              │
           └── 循环 ──────┘
           
流程：
Maker写 → Checker审 → 有问题？→ 返回Maker改
                       → 没问题？→ 合并部署
```

**为什么Maker和Checker必须分开？**

```
同一双眼睛看自己写的代码 → "写得不错啊" → 漏掉bug
不同的眼睛看同一段代码 → "这里有个边界情况没处理" → 抓住bug
```

### 🧠 经典模式二：Orchestrator-Worker（管弦乐指挥模式）

```
┌─────────────────────────────────────┐
│         Orchestrator Agent          │
│  （理解目标 → 拆解任务 → 分配任务）  │
└──┬──────────┬──────────┬───────────┘
   │          │          │
   ▼          ▼          ▼
Worker 1   Worker 2   Worker 3
(搜索)     (分析)     (写作)
   │          │          │
   └──────────┴──────────┘
          │
          ▼
   Orchestrator 汇总结果
          │
          ▼
       输出最终答案
```

### 🧠 经典模式三：Debate/Review（辩论模式）

```
Agent A（正方）  ←→  Agent B（反方）
  "应该用PostgreSQL"    "应该用MongoDB"
  
流程：
  A: 提出论点 → 给出论据
  B: 反驳论点 → 给出反论据
  A: 回应反驳 → ...
  ...
  达到共识或输出辩论摘要

  → 人类/评估Agent最终决策
```

**这个模式的真正价值**：LLM的"确认偏误"（confirmation bias）非常严重。辩论模式强制它看到相反观点。

### 🧠 多Agent团队的架构选择

| 模式 | 适用场景 | 优点 | 缺点 |
|:-----|:---------|:-----|:-----|
| **Maker-Checker** | 代码生成、内容创建 | 质量高 | 耗时加倍 |
| **Orchestrator-Worker** | 复杂多步任务 | 并行高效 | 需要好的拆解能力 |
| **Debate** | 决策/评估 | 减少偏见 | 可能陷入死循环 |
| **Pipeline** | 流水线式任务 | 简单可靠 | 灵活性差 |
| **Marketplace** | 开放式协作 | 高度灵活 | 管理成本高 |

> ✋ **费曼自测**：为什么"Maker-Checker"是一切多Agent系统的基础？请用"出版一本书"的流程做类比来解释。

### 🧠 多Agent系统的三大挑战

| 挑战 | 描述 | 缓解方案 |
|:-----|:-----|:---------|
| **通信成本** | Agent之间交换信息消耗大量Token | 限制通信频率，只传递关键信息 |
| **协调复杂性** | 多个Agent互相等待/冲突 | 清晰的层级结构 + 超时机制 |
| **一致性** | 各个Agent的输出风格/逻辑不一致 | 统一的系统指令 + 汇总Agent校对 |

### 📌 连线思考

多Agent不是"为了炫技"。它的核心洞察是：

> **一个模型的"智能"是有限的，但多个模型协作的"智能"可以突破单模型的极限。**

就像人类：一个天才的局限，可以被一群普通人的协作超越。

### ✅ 费曼三句话（闭眼自测）

1. **AI Agent和LLM聊天的本质区别是什么？** ——LLM聊天是"你问我答"，Agent是"你给我目标，我自己规划、执行、调整、完成"；Agent = LLM + 工具 + 记忆 + 规划。
2. **ReAct为什么是Agent的基础模式？** ——因为"思考→行动→观察→再思考"的循环让AI能边试边学；对比"一次想好所有步骤再行动"更灵活，能应对意外情况。
3. **多Agent系统为什么比单Agent更强？** ——单一Agent有"确认偏误"（自己写的代码自己觉得好），Maker-Checker分离解决了这个问题；专业化分工让每个Agent只做自己最擅长的事。

## 🔧 补充专题：Hermes Agent——开源多平台 Agent 框架实战

> 🎯 本专题介绍 Hermes Agent——NousResearch 推出的开源多平台 Agent 框架。它的核心理念是：**Agent 不应被锁死在单一平台**。Hermes Agent 实现了跨平台工具调用、Kanban 多 Agent 协作、目标持久化等创新特性，是 Day18 理论的"可运行版本"。

---

### 1. Hermes Agent 是什么？——Agent 界的"操作系统"

如果说 ReAct 是 Agent 的"设计模式"，那么 Hermes Agent 就是 Agent 的"运行时环境"。

**Hermes Agent 的核心理念：**

```
传统 Agent：
  一个模型 + 固定工具集 → 在单一平台上运行

Hermes Agent 的视角：
  Agent 应该像操作系统一样——
  管理资源（文件/API/浏览器）
  调度任务（多 Agent 协作）
  持久化状态（跨会话记忆）
  安全隔离（权限控制）
```

**与 Day18 四要素的对应关系：**

| Agent 四要素 | Hermes Agent 实现 | 说明 |
|:-------------|:-----------------|:-----|
| LLM | 支持 Hermes/LLaMA/GPT/Qwen 等多种模型 | 模型无关 |
| Tools | 跨平台工具集（Shell/文件/浏览器/API） | 统一接口 |
| Memory | 会话记忆 + `/goal` 持久化目标 | 双层记忆 |
| Planning | Task Kanban Board（待办/进行中/完成） | 可视化规划 |

---

### 2. Kanban 多 Agent 协作——"团队"而非"单兵"

Hermes Agent 最独特的创新是**看板系统（Kanban Board）**：

```
┌─────────────────────────────────────────────────────┐
│                   Task Kanban Board                   │
├──────────┬──────────────┬──────────────┬──────────────┤
│  Backlog  │   To Do      │   In Progress │   Done       │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 调研 API   │ 编写文档     │ 实现核心逻辑  │ 环境配置完成  │
│ 性能测试   │ 修复 Bug #12 │              │ 单元测试     │
│           │ 代码审查     │              │              │
└──────────┴──────────────┴──────────────┴──────────────┘
```

**多 Agent 协作模式：**

```
用户下达一个复杂任务 →
  Hermes Agent 拆解为多个子任务 →
    分配到 Kanban Board →
      多个 Agent 实例并行处理 →
        互相依赖的任务自动等待 →
          全部完成后合并结果 → 返回用户
```

**Agent 间通信机制：**

```
Agent A（编码） → 写入 shared/kernel.py → Agent B（测试）读取
                      ↑                          │
                      │        Agent C（审查）←────┘
                      │              │
                      └──────────────┘
                      （代码审查结果）
```

这种基于**共享文件系统**的通信方式，比传统的"Agent 之间发消息"更简单、更可靠、更容易调试——因为文件系统本身就是最好的"消息队列"。

### 3. `/goal`——目标持久化与记忆系统

Hermes Agent 的 `/goal` 机制解决了 Agent 的"金鱼记忆"问题：

```
用户："/goal 帮我每天检查 GitHub Issue，自动回复已知 Bug"

Hermes Agent 的处理：
1. 解析目标 → "每天检查 GitHub Issue，自动回复已知 Bug"
2. 分解子目标 →
   ├── 每 24 小时检查新的 Issue
   ├── 用历史记录判断是否已知 Bug
   └── 如果是已知 Bug → 自动回复解决方案
3. 持久化存储 → 保存到 goals.json
4. 定时触发 → 即使 Agent 重启也不会丢失
```

**记忆系统的三层结构：**

```
Hermes Agent Memory
├── 工作记忆（Session）
│   ├── 当前对话上下文
│   └── 当前任务状态
├── 目标记忆（Persistent）
│   ├── 用户设定的长期目标
│   └── 每个目标的进度和状态
└── 知识记忆（File-based）
    ├── 项目代码（向量化索引）
    ├── 过往对话（检索式）
    └── 工具使用记录（经验积累）
```

### 4. 安全体系——让 Agent"可控"

Agent 能力的增强也意味着风险的增加。Hermes Agent 内置多层安全机制：

**权限分级系统：**

```
Level 0：只读
├── 可操作：读取文件、查看 API
└── 不可操作：修改文件、执行命令

Level 1：标准
├── 可操作：读写文件、运行已知命令
└── 不可操作：安装软件、修改系统配置

Level 2：超级
├── 可操作：全部操作
└── 需要：用户明确确认
```

**沙箱机制：**

```
任务 → 沙箱评估
    ├── 无害 → 直接执行
    ├── 敏感 → 请求用户确认
    └── 危险 → 拒绝执行
```

| 安全特性 | 说明 | 对应现实场景 |
|:---------|:-----|:------------|
| 权限分级 | 按操作风险等级自动限制 | 员工门禁卡(办公室/机房/保险库) |
| 沙箱评估 | 自动判断操作风险等级 | 机场安检(随身/托运/禁止) |
| 命令白名单 | 只允许预授权的命令执行 | 外卖App只能点餐不能转账 |
| 文件系统隔离 | Agent 只能在特定目录操作 | 访客只能进客厅不能进卧室 |
| `confirm` 命令 | 需要确认 Agent 的下一步操作 | "需要确认后再执行吗？" |

### 5. 实战场景——用 Hermes Agent 构建"自动 PR 审查系统"

**场景描述：** GitHub PR 来了 → Agent 自动审查代码 → 运行测试 → 生成审查报告 → 回复 PR

**传统的做法（没有 Agent）：**
```
开发者收到 PR 通知 → 手动 pull 分支 → 读代码 → 运行测试
→ 写 review 评论 → 耗时 30 分钟
```

**用 Hermes Agent 的做法：**

```
用户输入："/goal 自动审查 main 仓库的 PR"

Hermes Agent 创建一个 Kanban 任务队列：
┌──────────────────────────────────────────────┐
│ PR 审查 Pipeline                             │
├──────────┬──────────┬────────────┬────────────┤
│ 待处理    │ 进行中   │ 待审核     │ 已完成     │
├──────────┼──────────┼────────────┼────────────┤
│ PR #456  │ PR #455  │            │ PR #453 ✓  │
│ PR #457  │          │            │ PR #454 ✓  │
└──────────┴──────────┴────────────┴────────────┘

每个 PR 的审查流程：
1. Agent A（GitHub）→ 读取 PR diff 和上下文
2. Agent B（代码分析）→ 静态分析、安全检查
3. Agent C（测试）→ 运行测试套件
4. Agent D（汇总）→ 整合所有结果
5. Agent E（回复）→ 在 PR 中发表审查意见
```

**Agent 间协作的代码逻辑（伪代码）：**

```
# Agent A：读取 PR
pr_diff = github.get_pr_diff("repo", 455)

# Agent B：代码分析（接在 Agent A 完成后自动触发）
issues = code_analyzer.analyze(pr_diff)
security_issues = security_scanner.scan(pr_diff)

# Agent C：运行测试
test_results = test_runner.run()

# Agent D：汇总（等待 B 和 C 都完成）
report = generate_report(issues, security_issues, test_results)

# Agent E：回复
github.post_comment("repo", 455, report)
```

这个流程完美体现了 Day18 讲解的 **ReAct 循环 + 多 Agent 协作 + 工具调用**——只不过从"理论"变成了"可运行的代码"。
