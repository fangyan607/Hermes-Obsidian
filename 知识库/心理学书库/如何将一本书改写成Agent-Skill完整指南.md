## 一、概述：什么是 Skill

### 1.1 Skill 的本质

Skill 是 **"给 AI 的新人入职手册"**——它将领域知识转化为 AI 可执行的程序化指令。

```
传统知识（书）→ 被动阅读 → 依赖读者理解能力
Skill（程序化知识）→ 主动执行 → AI 按步骤行动
```

### 1.2 Skill 与书的区别

| 维度 | 书 | Skill |
|------|---|-------|
| 知识形态 | 叙述性、解释性 | 指令性、可执行 |
| 使用方式 | 人类阅读理解 | AI 直接执行 |
| 结构特点 | 章节递进、案例丰富 | 流程驱动、决策树清晰 |
| 目标 | 理解作者观点 | 完成具体任务 |
| 迭代方式 | 重读、笔记 | 修改指令、测试反馈 |

### 1.3 什么样的书适合改写成 Skill

**适合**：
- ✅ 方法论/框架类书籍（如《金字塔原理》《思考，快与慢》）
- ✅ 流程清晰的实操指南（如写作方法、设计方法论）
- ✅ 有明确步骤的教程类书籍
- ✅ 决策框架类书籍

**不适合**：
- ❌ 纯叙事类（小说、传记）
- ❌ 信息汇编类（百科、词典）
- ❌ 观点表达类（随笔、评论集）
- ❌ 过于抽象的哲学著作

### 1.4 Skill 的目录结构

```
skill-name/
├── SKILL.md                    # 核心指令文件（必须）
│   ├── YAML frontmatter        # 元数据（必须）
│   │   ├── name:               # Skill 名称
│   │   └── description:        # 触发描述
│   └── Markdown 内容            # 执行指令
│
└── bundled-resources/          # 打包资源（可选）
    ├── scripts/                # 可执行脚本
    ├── references/             # 参考文档
    └── assets/                 # 模板、示例文件
```

## 二、前置准备

### 2.1 选择目标书籍

**评估标准**：

| 评估维度 | 高适配 | 中适配 | 低适配 |
|---------|--------|--------|--------|
| 方法论清晰度 | 有明确步骤框架 | 有原则但步骤模糊 | 只有观点没有方法 |
| 可操作性 | 每章有练习/检查清单 | 有案例但无练习 | 纯理论阐述 |
| 决策节点 | 明确的"如果...则..." | 隐含的决策逻辑 | 无决策判断 |
| 普适性 | 跨领域适用 | 特定领域但有迁移价值 | 极度特定场景 |

### 2.2 准备工作环境

```bash
# 1. 创建 Skill 目录
mkdir -p ~/.claude/skills/[book-name]-method/SKILL.md
mkdir -p ~/.claude/skills/[book-name]-method/{scripts,references,assets}

# 2. 准备素材目录
mkdir -p ~/book-to-skill/[book-name]/{notes,extractions,examples}
```

### 2.3 准备阅读工具

- **电子版书籍**：方便搜索和摘录
- **笔记工具**：用于记录关键框架
- **思维导图工具**：可视化书籍结构

## 三、Phase 1: 书籍分析与解构

### 3.1 第一遍阅读：结构扫描

**目标**：建立书籍的整体认知地图

**操作步骤**：

1. **阅读目录**，绘制书籍结构图
2. **识别核心章节**（通常占全书 60% 价值）
3. **标注方法论密集区**（有步骤、清单、框架的部分）

**输出模板**：

```markdown
## 书籍结构扫描

### 基本信息
- 书名：
- 作者：
- 出版年份：
- 核心主题：

### 章节价值评估

| 章节 | 核心内容 | 方法论密度 | Skill 价值 |
|------|---------|-----------|-----------|
| 第1章 | ... | 高/中/低 | 核心框架/补充/忽略 |
| 第2章 | ... | ... | ... |
| ... | ... | ... | ... |

### 核心方法论章节（选 3-5 章）
1. 第X章：[章节名] - [一句话价值]
2. ...
```

### 3.2 第二遍阅读：框架提取

**目标**：提取可程序化的方法论框架

**提取重点**：

| 提取类型 | 说明 | 示例 |
|---------|------|------|
| **核心概念** | 书籍独创或强调的概念 | "金字塔结构"、"SCQA 模型" |
| **流程步骤** | 有序的行动序列 | "写作五步法" |
| **决策规则** | 条件-行动映射 | "如果信息量大，则使用..." |
| **检查清单** | 验证标准 | "好标题的 4 个标准" |
| **模板** | 可复用的结构 | "论证结构模板" |
| **案例** | 方法的具体应用 | 示例文章分析 |

**输出模板**：


[模板内容]

## 四、Phase 2: 框架提炼与转化

### 4.1 从叙述到指令的转化原则

**核心转变**：

```
书中原文：「在写作时，作者应该先思考读者的需求...」

Skill 指令：
「收到写作任务后，第一步：分析读者画像
  - 读者是谁？（角色、背景）
  - 读者想知道什么？（核心问题）
  - 读者已有知识？（起点）
  输出：读者画像卡片」
```

### 4.2 方法论的层次化重构

**三层结构**：

```
┌─────────────────────────────────────┐
│  Layer 1: 核心原则                   │
│  「为什么这样做」                     │
├─────────────────────────────────────┤
│  Layer 2: 执行流程                   │
│  「怎么做」                           │
├─────────────────────────────────────┤
│  Layer 3: 操作细节                   │
│  「具体步骤和判断标准」                │
└─────────────────────────────────────┘
```

**转化示例**：

```markdown
## 原书内容（Layer 1-2）

「金字塔原理的核心是：结论先行，自上而下表达。
先说结论，再说支撑论点，最后是事实依据。」

## Skill 转化（Layer 3）

### 执行流程

**Step 1: 提炼核心结论**
- 输入：用户的原始内容
- 动作：用一句话概括核心观点
- 标准：不超过 20 字，包含主语+谓语+宾语
- 输出：`核心结论：[一句话]`

**Step 2: 分解支撑论点**
- 条件判断：
  - 如果是因果关系 → 提取 3-5 个原因
  - 如果是分类关系 → 按 MECE 原则分解
  - 如果是过程关系 → 提取关键步骤
- 标准：每个论点独立且支撑结论
- 输出：`论点1: ...` `论点2: ...` ...

**Step 3: 补充事实依据**
- 动作：为每个论点找 1-3 个支撑事实
- 来源：用户提供的数据、案例、引用
- 输出：`依据1.1: ...` ...

**Step 4: 检查金字塔结构**
- [ ] 结论是否在开头？
- [ ] 论点是否支撑结论？
- [ ] 论点之间是否 MECE？
- [ ] 依据是否充分？
```

### 4.3 决策树的构建

**将书的判断逻辑转化为决策树**：

```markdown
## 决策树：选择文章结构

```
开始
  │
  ▼
问题类型？
  │
  ├─ 解释「是什么」→ 定义式结构
  │    └─ 概念定义 → 特征说明 → 举例
  │
  ├─ 说明「怎么做」→ 流程式结构
  │    └─ 目标 → 步骤1 → 步骤2 → ... → 检查
  │
  ├─ 论证「为什么」→ 论证式结构
  │    └─ 观点 → 论据1 → 论据2 → ... → 结论
  │
  └─ 比较「选哪个」→ 对比式结构
       └─ 维度1对比 → 维度2对比 → ... → 建议
```
```

### 4.4 检查清单的程序化

**原书检查清单**：

```markdown
好标题的标准：
- 简洁
- 准确
- 有吸引力
- 与内容相关
```

**Skill 转化**：

```markdown
## 标题质量检查

**执行时机**：生成标题后

**检查流程**：

1. **简洁性检查**
   - 标准：标题长度 ≤ 20 字
   - 动作：如果超标，删减修饰词

2. **准确性检查**
   - 标准：标题反映核心内容
   - 动作：对比标题与首段主题

3. **吸引力检查**
   - 标准：包含好奇心触发词 或 利益点
   - 好奇心触发词：如何、为什么、竟然、秘密
   - 利益点词汇：快速、简单、有效

4. **相关性检查**
   - 标准：标题关键词在正文出现 ≥ 3 次
   - 动作：检查关键词密度

**输出格式**：
```
标题检查结果：
- 简洁性：✅/❌ [当前字数]
- 准确性：✅/❌ [匹配度]
- 吸引力：✅/❌ [触发词/利益点]
- 相关性：✅/❌ [关键词次数]

建议修改：[具体建议]
```
```

- [ ] 检查清单有明确的判断标准
- [ ] 流程有输入/输出定义

### 5.2 编写 YAML Frontmatter

**name 命名规范**：
- 使用英文小写
- 多个单词用连字符连接
- 格式：`[核心方法]-[类型]`
- 示例：`pyramid-writing`, `first-principles-thinking`

**description 编写规范**：
- 第一句：描述功能
- 第二句：触发条件
- 第三句：核心价值

**示例**：

```yaml
---
name: pyramid-writing
description: |
  金字塔写作法：将复杂信息组织成逻辑清晰的分层结构。
  触发场景：需要写作/整理思路/制作演示文稿/汇报工作。
  核心价值：结论先行、自上而下、MECE 分解。
---
```

### 5.3 编写执行流程

**关键原则**：

1. **动词开头**：用祈使句描述动作
   - ✅ "分析用户需求"
   - ❌ "需要分析用户需求"

2. **明确输入输出**：
   ```markdown
   **输入**：用户提供的原始文本
   **输出**：结构化的大纲
   ```

3. **条件判断清晰**：
   ```markdown
   **条件判断**：
   - 如果信息量 < 3 个要点 → 使用简单结构
   - 如果信息量 ≥ 3 个要点 → 使用金字塔结构
   ```

4. **包含失败处理**：
   ```markdown
   **异常处理**：
   - 如果无法提炼结论 → 提示用户提供更多信息
   - 如果信息矛盾 → 标注矛盾点，请求澄清
   ```

## 六、Phase 4: 质量验证

### 6.1 功能测试

**测试类型**：

| 测试类型 | 方法 | 通过标准 |
|---------|------|---------|
| **正向测试** | 给标准输入，检查输出 | 输出符合预期格式和内容 |
| **边界测试** | 给极端/异常输入 | 能优雅处理或给出明确提示 |
| **回归测试** | 修改后重跑已有测试 | 结果一致 |

**测试用例模板**：

```markdown
## 测试用例 [ID]

### 输入
```
[测试输入]
```

### 期望输出
```
[期望输出]
```

### 实际输出
```
[实际输出]
```

### 结果
✅ 通过 / ❌ 失败

### 问题分析
[如果失败，分析原因]
```

### 6.2 质量检查清单

```markdown
## Skill 质量检查清单

### 完整性检查
- [ ] 有明确的触发条件
- [ ] 执行流程覆盖完整任务
- [ ] 有输入输出定义
- [ ] 有异常处理说明

### 可操作性检查
- [ ] 每个步骤都是可执行的动词指令
- [ ] 判断条件有明确的标准
- [ ] 输出格式有具体要求
- [ ] 有示例演示

### 准确性检查
- [ ] 与原书方法论一致
- [ ] 没有过度简化或曲解
- [ ] 案例与原书一致或有来源

- [ ] 结构清晰，易于理解
```

### 6.3 用户测试（可选）

**测试流程**：

1. 找 2-3 个目标用户
2. 给他们任务，让他们用 Skill 完成
3. 收集反馈：
   - 是否理解执行流程？
   - 是否遇到歧义？
   - 输出是否符合预期？
4. 迭代改进

### 6.4 迭代优化

**优化优先级**：

```
P0: 导致无法正确执行的问题
```

## 十、常见问题 FAQ

### Q1: 书太厚了，怎么简化？

**策略**：
1. 先识别核心方法论（通常 3-5 个）
2. 只保留与核心方法直接相关的内容
3. 案例选择最有代表性的 1-2 个
4. 详细内容放到 references 目录

### Q2: 书的方法论有多个流派怎么办？

**策略**：
1. 选择原书主推的流派作为核心
2. 在"决策规则"中标注流派选择条件
3. 在 references 中说明流派差异

### Q3: 怎么处理书中的图表？

**策略**：
1. 图表内容转化为文字描述
2. 流程图转化为决策树或步骤列表
3. 复杂图表放到 references 或 assets

### Q4: Skill 太长了怎么办？

**策略**：
1. SKILL.md 只保留执行流程和核心规则
2. 详细说明移到 references
3. 按功能拆分为多个子 Skill

### Q5: 如何处理书中的个人观点？

**策略**：
1. 区分"方法论"和"观点"
2. 方法论转化为执行指令
3. 观点在"局限性"中标注

## 结语

将一本书改写为 Skill，本质上是**知识的工程化**：

```
作者的知识 → 书的叙述 → 你的理解 → 程序化指令 → AI 执行
```

这个过程要求你：
1. **深度理解**原书方法论
2. **抽象提炼**核心框架
3. **具体转化**为可执行步骤
4. **持续迭代**优化效果

一个好的 Skill，让书中知识从"读过"变成"用过"，从"理解"变成"行动"。
