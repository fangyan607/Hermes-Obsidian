# Day17：Context Engineering & RAG——让AI拥有知识（番茄81-85）

> **Part 4 · 大语言模型与智能体**
> **悬疑探案进度**：LLM的大脑你了解了，对话的艺术你掌握了，现在——给它装一个"外接硬盘"。
> **番茄范围**：🍅81-85

---

## 悬疑开场：一个失忆的天才

想象一下这个场景：

你有一个朋友，他是世界上最博学的人。他读过所有图书、所有论文、所有维基百科——但他有个问题：

**他的知识停留在2025年1月。**

你问他"今天天气怎么样？"——他无法回答（因为他不联网）。
你问他"我们公司最新季度的财报怎么样？"——他不知道（因为那是内部数据）。
你问他"我刚才说了什么？"——他忘了（因为他的记忆只有128K token）。

这个朋友，就是今天的大语言模型。

**你该怎么让他"知道"他不知道的东西？**

方法1：重新训练他（太贵了，一次训练花几千万美元）。
方法2：把资料写到他的Prompt里（Context窗口有限，装不下太多）。
方法3：**给他装一个"图书馆"——他知道怎么查，不用全记住。**

方法3就是RAG（Retrieval-Augmented Generation，检索增强生成）。

这是一个比"让模型更聪明"更实际、更便宜的方案。今天，你将学会它。

## 🍅 番茄81 · T1：Context Window——Token的战场

### 🔍 犯罪现场：为什么"塞更多内容"不是万能药？

既然模型不知道最新知识，那就把最新知识写在Prompt里吧——对吧？

这就是Context Window的作用。但这件事远没有想象的那么简单。

### 🧠 什么是Context Window？

**Context Window（上下文窗口）** = LLM一次能"看到"的文本量。

```
GPT-3 (2020):    2K tokens  ≈ 1,500字
GPT-3.5 (2022):  4K tokens  ≈ 3,000字  
GPT-4 (2023):    8K→32K tokens  ≈ 24,000字
Claude 2 (2023): 100K tokens  ≈ 75,000字（《三体》三部曲的1/3）
Gemini 1.5 (2024): 1M tokens  ≈ 750,000字（整套《三体》）
GPT-4o (2024):   128K tokens
Claude 4 (2025): 200K+ tokens
```

### 🧠 "窗口越大越好"——但有一个隐藏陷阱

**你需要知道的残酷真相**：

```
Context窗口越大 → 能塞的信息越多 → 但"找信息"越困难

就像：你的办公桌越大 → 能堆的文件越多 → 但找到某个文件越花时间
```

研究表明（Liu et al., 2023; Anthropic, 2024）：

| Context中的信息位置 | LLM的"注意度" |
|:-------------------|:-------------|
| 开头（Primacy） | ⭐⭐⭐ 高 |
| 结尾（Recency） | ⭐⭐⭐ 高 |
| 中间 | ⭐ 低（"迷失在中间"） |

**"迷失在中间"（Lost in the Middle）**：当你的关键信息放在Context窗口中间时，模型很容易忽略它。

> ✋ **费曼自测**：用图书馆借书的类比解释"迷失在中间"现象。为什么越大的Context不一定越好？

### 🧠 Context管理的三种策略

| 策略 | 做法 | 适用场景 |
|:-----|:-----|:---------|
| ① 直接塞 | 把所有内容放在开头或结尾 | 内容少（< 4K tokens） |
| ② 总结压缩 | 先让AI总结长文本，再把摘要放进去 | 长内容但不需要细节 |
| ③ 检索+注入 | **只放最相关的那部分** | 海量内容需要精确查询 |

策略③就是RAG的核心思想。下一番茄详谈。

### 🧠 Token计费现实

别忘了——Token就是钱。

```
GPT-4o:    $2.50/1M输入token  |  $10.00/1M输出token
Claude 4:  $3.00/1M输入token  |  $15.00/1M输出token

如果你每次请求塞入100K token上下文：
每次输入成本 ≈ $0.25-$0.30
每天10,000次请求 ≈ $2,500-$3,000/天
```

**Context Engineering不是"我能塞多少"，而是"我该塞多少才够用"。**

## 🍅 番茄82 · T2：RAG架构——检索→增强→生成：三阶段拆解

### 🔍 犯罪现场：当一个LLM"查了资料再回答"

RAG不是一个模型，而是一个**系统架构**。

它的核心流程只有三步，但每一步都有深度。

### 🧠 RAG三阶段全景

```
用户问："我们公司2024年Q4的营收是多少？"
                │
    ┌────────────▼────────────┐
    │  ① 检索（Retrieve）      │
    │  在知识库中找到相关文档   │
    │  如：2024年报.pdf → 第23页 │
    └────────────┬────────────┘
                 │
    ┌────────────▼────────────┐
    │  ② 增强（Augment）       │
    │  把检索到的文档+用户问题  │
    │  合并成一个"增强版"Prompt  │
    └────────────┬────────────┘
                 │
    ┌────────────▼────────────┐
    │  ③ 生成（Generate）      │
    │  LLM基于增强后的Context   │
    │  生成最终答案            │
    └────────────┬────────────┘
                 │
  输出："2024年Q4营收为42.5亿美元，
         同比增长18%，主要得益于..."
```

### 🧠 ① 检索（Retrieve）——找到相关文档

**朴素方法**：关键词搜索（BM25）。

```
用户搜"Q4营收" → 找包含"Q4"和"营收"的文档 → 返回
优点：简单、快
缺点：同义词问题（"Q4"="第四季度"、"营收"="收入"）
```

**进阶方法**：向量检索（语义搜索）。

```
用户搜"Q4营收"
→ 把问题转成向量 [0.12, -0.45, 0.78, ...]
→ 在知识库找"最相似"的向量
→ 返回["2024年报.pdf", "CFO季度总结.pdf"]
优点：理解语义（"营收"≈"收入"）
```

### 🧠 ② 增强（Augment）——把碎片拼成完整Prompt

```
原始Prompt：
  用户：2024年Q4营收是多少？

增强后Prompt：
  系统：用以下文档回答用户问题。
  
  [文档1]：2024年报第23页
  "第四季度营收达到42.5亿美元，同比增长18%..."
  
  [文档2]：CFO季度总结
  "Q4增长主要来自亚太区新客户..."
  
  用户：2024年Q4营收是多少？
```

**关键设计问题**：
- 放多少文档？——太多会"迷失在中间"，太少可能漏信息
- 怎么排序？——最相关的放开头和结尾
- 原文档要不要保留格式？——表格、列表可能会丢失

### 🧠 ③ 生成（Generate）——让LLM基于引用回答

```
最终输出：
  "根据2024年报，公司Q4营收为42.5亿美元，同比增长18%。
  主要增长动力来自亚太区新客户（CFO季度总结）。
  
  数据来源：
  - 2024年报第23页
  - CFO 2024年Q4季度总结"
```

**这个"引用源"是关键**——没有来源的RAG回答，就是虚构。

> ✋ **费曼自测**：用"开卷考试"做类比解释RAG的三个阶段。学生在考试中翻书找答案 → 对应哪个阶段？

### 🧠 RAG vs 微调：什么时候用什么？

| 维度 | RAG | Fine-tuning |
|:-----|:----|:------------|
| **成本** | 低（按需检索） | 高（训练成本） |
| **知识更新** | 秒级（更新知识库即可） | 慢（需要重新训练） |
| **知识类型** | 事实性、时效性知识 | 技能、风格、行为模式 |
| **幻觉风险** | 较低（有源可查） | 较高（可能"记住"错误信息） |
| **隐私** | 好（知识不出本地） | 需谨慎（数据用于训练） |

**黄金法则**：
- **RAG**：让AI知道"你不知道的"（最新新闻、内部文档、私人数据）
- **Fine-tuning**：让AI学会"你不会的"（特定写作风格、领域术语、任务技能）

## 🍅 番茄83 · T3：向量数据库——Embedding+相似度搜索

### 🔍 犯罪现场：计算机怎么知道"苹果"≈"水果"但≠"手机"？

关键词搜索告诉你"苹果手机"包含"苹果"。

语义搜索告诉你"苹果"可能指水果、公司或手机——**看你问什么**。

语义搜索的秘密，就是Embedding。

### 🧠 什么是Embedding？

**Embedding = 把文本转成一组数字（向量）。**

```
"天空是蓝色的" → [0.23, -0.56, 0.89, -0.12, 0.45, ...]
"大海是蓝色的" → [0.25, -0.53, 0.87, -0.10, 0.42, ...]  ← 相似
"我喜欢编程"   → [-0.31, 0.67, -0.22, 0.55, -0.78, ...]  ← 不相似
```

**关键**：语义相似的文本，向量距离也近。

就像在地图上：北京和上海离得近，北京和巴黎离得远。

### 🧠 向量搜索的工作原理

```
Step 1：把知识库所有文档转成向量
  doc1 = "2024年营收42.5亿" → vec1 = [0.1, -0.2, 0.8, ...]
  doc2 = "2025年目标50亿"   → vec2 = [0.1, -0.1, 0.9, ...]
  doc3 = "公司团建通知"     → vec3 = [-0.5, 0.7, -0.3, ...]

Step 2：把用户问题转成向量
  问题："去年的营收是多少？" → q_vec = [0.12, -0.18, 0.82, ...]

Step 3：计算距离，取Top-K
  doc1: 距离 0.05 ← 最相关
  doc2: 距离 0.15
  doc3: 距离 1.25 ← 不相关
```

### 🧠 主流向量数据库对比

| 产品 | 类型 | 特点 | 适用场景 |
|:-----|:-----|:-----|:---------|
| **Pinecone** | 云服务 | 托管、高性能、上手快 | 生产环境 |
| **Weaviate** | 开源+云 | 自带Schema、混合搜索 | 需要结构化+向量 |
| **Qdrant** | 开源 | Rust编写、极致性能 | 高吞吐场景 |
| **Milvus** | 开源 | 分布式、大规模 | 十亿级向量 |
| **Chroma** | 开源嵌入式 | 轻量、Python原生 | 原型开发、学习 |
| **pgvector** | PostgreSQL扩展 | 和现有DB集成 | 不想加新中间件 |
| **LanceDB** | 嵌入式 | 基于Lance列式格式 | 数据科学工作流 |

> ✋ **费曼自测**：用"找朋友"做类比解释向量搜索——你在人群中找一个"看起来像程序员"的人，你是怎么判断的？

### 🧠 Embedding模型的进化

| 模型 | 维度 | 特点 |
|:-----|:-----|:-----|
| OpenAI ada-002 | 1536 | 通用性强，API调用 |
| text-embedding-3-large | 3072 | 更强，更贵 |
| Cohere embed | 4096 | 企业级 |
| BGE (BAAI) | 1024 | 开源，中文友好 |
| E5 (微软) | 1024 | 开源，多语言 |
| GTE (阿里) | 1024 | 中英文均可 |

**选择Embedding模型的关键**：
1. 你的文档是什么语言？→ 选对应语言擅长的模型
2. 你的维度需求？→ 维度越高 = 越精准 = 越贵
3. 成本敏感？→ 开源模型免费部署

## 🍅 番茄84 · T4：高级RAG——HyDE、RAPTOR、Self-RAG

### 🔍 犯罪现场：基础RAG不够聪明怎么办？

基础RAG的问题：

- **问题**："苹果公司2024年的MacBook Pro比上一代快了多少？"
- **检索**：找到了包含"苹果"和"MacBook Pro"的文档...
- **结果**：有5篇关于MacBook Pro的文档，有的是2023款的，有的是2024款的——**哪个是你要的？**

基础RAG的"检索"太粗糙了。你需要更智能的方法。

### 🧠 高级RAG技术一：HyDE（Hypothetical Document Embedding）

**核心思想**：先让LLM"想象"一个理想答案，再用这个想象去搜索。

```
传统RAG：
  问题 → 向量化问题 → 搜索相似文档 → 生成答案

HyDE：
  问题 → LLM生成"假设的理想文档" → 向量化假设文档 → 搜索相似
                                                              ↑
  优势：假设文档和真实文档的"语义距离"比问题和文档更近
```

**类比**：你去图书馆找书，与其描述你想要的书（"一本关于...的书"），不如让专家先写一段类似书的摘要，再找和这段摘要最像的书。

### 🧠 高级RAG技术二：RAPTOR（递归摘要+树状检索）

**核心思想**：文档分层——从小块到整篇，逐级摘要，构建一棵"知识树"。

```
                      [根摘要]
                      /      \
               [章节摘要]    [章节摘要]
               /    |    \    /    |    \
            [小节] [小节] [小节] [小节] [小节] [小节]
            / \    / \    / \    / \    / \    / \
          [块][块][块][块][块][块][块][块][块][块][块][块]

  检索策略：
  用户问宏观问题 → 搜根节点（摘要）
  用户问具体问题 → 搜叶子节点（原文块）
  用户问中等问题 → 搜索中间节点
```

**优势**：不同粒度的信息放在不同层级，检索精度和召回率双高。

### 🧠 高级RAG技术三：Self-RAG

**核心思想**：让LLM自己决定"要不要检索、检索什么、是否相信检索结果"。

```
与传统RAG的根本区别：
  传统RAG：每次必检索 → 增强 → 生成
  Self-RAG：LLM判断"我需要更多信息吗？" → 需要才检索 → 评估结果可信度 → 生成

  决策点：
  1. 检索判断：这个问题我的知识够吗？
  2. 段落评估：这段检索结果相关吗？
  3. 可信度判断：这个信息可信吗？
  4. 生成选择：用检索结果还是用自己的知识？
```

**类比**：传统RAG是"考试时每道题都翻书"，Self-RAG是"只在不确定时才翻书，而且翻到答案后还要判断这答案对不对"。

### 🧠 高级RAG技术对比

| 技术 | 解决什么问题 | 核心创新 | 复杂度 |
|:-----|:------------|:---------|:------:|
| **HyDE** | 问题和知识库"语言"不一致 | 用假设答案做桥梁 | ⭐⭐ |
| **RAPTOR** | 信息粒度不匹配 | 递归摘要+树状检索 | ⭐⭐⭐⭐ |
| **Self-RAG** | 检索过多/过少 | LLM自判断检索需求 | ⭐⭐⭐ |
| **GraphRAG** | 多跳关系推理 | 知识图谱+向量检索 | ⭐⭐⭐⭐⭐ |
| **Corrective RAG** | 检索结果质量差 | 检索后自动修正 | ⭐⭐ |

> ✋ **费曼自测**：Self-RAG和"反思"有什么关系？用"客服查资料"做类比解释Self-RAG的三个决策点。

### 🧠 GraphRAG：当知识图谱遇上RAG

微软提出的GraphRAG是2024年最受关注的高级RAG变体：

```
传统RAG的局限："谁杀了张三？" → 检索不到直接答案 → 失败

GraphRAG的方案：
1. 把文档构建成知识图谱
   （张三）—[被...杀]→（李四）? 
   （张三）—[与...有仇]→（王五）
2. 检索时同时搜向量+图关系
3. 通过实体关系推理"间接答案"
```

## 🍅 番茄85 · T5：Context Engineering综合案例——构建一个知识问答系统

### 🔍 犯罪现场：把理论变成系统

学了这么多概念，是时候动手了——设计一个能回答"内部知识库问题"的RAG系统。

### 🧠 场景设定

**需求**：一家中型公司（500人）想要一个内部知识问答Bot，能回答关于：
- 公司政策（年假、报销、远程办公）
- 产品文档（API文档、用户手册、Release Notes）
- 项目信息（Sprint回顾、设计文档）

### 🧠 系统设计

```
                    ┌─────────────────────┐
                    │   用户问问题          │
                    │   "年假怎么申请？"    │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Query Understanding  │
                    │  • 问题分类           │
                    │  • 实体提取           │
                    │  • 意图识别           │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
    ┌─────▼──────┐      ┌─────▼──────┐      ┌─────▼──────┐
    │ 检索策略1   │      │ 检索策略2   │      │ 检索策略3   │
    │ 向量搜索    │      │ 关键词搜索  │      │ 图关系检索   │
    │ (语义相关)  │      │ (精准匹配)  │      │ (实体关系)  │
    └─────┬──────┘      └─────┬──────┘      └─────┬──────┘
          │                    │                    │
          └────────────────────┼────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  结果融合与排序      │
                    │  • 去重              │
                    │  • 按相关度排序       │
                    │  • 限制Top-K         │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Prompt组装          │
                    │  系统：用以下文档回答 │
                    │  [Doc1] [Doc2] [Doc3]│
                    │  用户：[问题]        │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  LLM生成+引用标注    │
                    │  "根据[文档A]，年假  │
                    │   需要提前3天在系统   │
                    │   提交申请..."      │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Validation         │
                    │  • 有引用吗？        │
                    │  • 引用对应吗？      │
                    │  • 如果没把握：拒答  │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │      最终答案        │
                    └─────────────────────┘
```

### 🧠 关键设计决策

#### 决策1：分块策略

将文档切成合适的"块大小"：

```
文档："2024年公司政策.pdf" (约50页)
  → 按章节切块（每个政策条款一块）
  → 每块500-1000字
  → 保留元数据（来源页码、章节名、发布日期）
```

#### 决策2：检索策略组合

| 问题类型 | 检索策略 | 权重 |
|:---------|:---------|:-----|
| 事实性（"年假几天？"） | 关键词 → 向量 | 60% + 40% |
| 概念性（"什么是弹性工作？"） | 向量 → 关键词 | 80% + 20% |
| 多跳（"新员工的年假政策和去年比有什么变化？"） | 向量 + 图关系 | Hybrid |

#### 决策3：安全与质量

```
安全策略：
  - 权限过滤：用户只能检索到有权限的文档
  - PII脱敏：检索结果中自动隐藏个人信息
  - 内容审核：生成答案后检查是否包含敏感内容

质量策略：
  - 源引用：每个事实必须标注来源
  - 不确定时拒绝："抱歉，我没有找到相关信息"
  - 反馈循环：用户可以对答案"点赞/点踩"
```

### 🧠 用代码看RAG核心逻辑（伪代码）

```python
def rag_query(user_question: str, k: int = 3):
    # Step 1: 问题理解
    intent = classify_intent(user_question)  # "policy" | "product" | "project"
    
    # Step 2: 检索
    query_vec = embed(user_question)  # 向量化
    vector_results = vector_db.search(query_vec, k=k)
    keyword_results = keyword_search(user_question, k=k)
    
    # Step 3: 融合
    combined = merge_and_rank(vector_results, keyword_results)
    top_docs = combined[:k]
    
    # Step 4: 增强
    system_prompt = f"""
    用以下文档回答问题。如果文档中没有相关信息，请说"没有找到相关信息"。
    每个事实必须标注来源文档。
    
    {format_docs(top_docs)}
    
    用户问题：{user_question}
    """
    
    # Step 5: 生成
    answer = llm.generate(system_prompt)
    
    # Step 6: 验证
    if not has_citations(answer):
        answer = "无法回答，请重新表述您的问题。"
    
    return answer
```

### 1. LangChain 是什么？——LLM 世界的"标准库"

### 2. LangChain RAG——从原型到生产的完整路径

**最小 RAG 示例（对比 Day17 的 RAG 原理）：**

```python
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain_openai import ChatOpenAI

# 1. 加载文档
loader = TextLoader("knowledge_base.txt")
documents = loader.load()

# 2. 分块
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
chunks = text_splitter.split_documents(documents)

# 3. 向量化 + 存储
vectordb = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings()
)

# 4. RAG Chain
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o"),
    chain_type="stuff",  # 也可用 map_reduce / refine / map_rerank
    retriever=vectordb.as_retriever(search_kwargs={"k": 3})
)

# 5. 使用
response = qa_chain.invoke("什么是 RAG？")
```

这 20 行代码实现了 Day17 讲解的一整套 RAG 流程。LangChain 的核心价值就在于此——**把论文级别的概念变成了 import 语句**。

**高级 RAG 策略在 LangChain 中的实现：**

| RAG 策略 | LangChain 组件 | 配置方式 |
|:---------|:--------------|:---------|
| HyDE | `HypotheticalDocumentEmbedder` | 先生成假设文档，再检索 |
| RAPTOR | `ParentDocumentRetriever` + 摘要 | 检索父文档块 + 子块 |
| Self-RAG | `SelfQueryRetriever` | 根据查询自我判断检索策略 |
| 多查询 | `EnsembleRetriever` | 多个检索器加权融合 |
| 重排序 | `ContextualCompressionRetriever` | 检索后压缩/重排 |

### 3. LangGraph——状态化 Agent 工作流

LangChain 的早期版本使用 Chain 来组合步骤，但 Chain 是**线性**的——A → B → C。真正的 Agent 系统需要**循环、分支、状态管理**。

**LangGraph 的诞生**就是为了解决这个问题。

```
Chain 模式（线性）：
Prompt → LLM → Parse → Done

Graph 模式（循环）：
         ┌─────────────────┐
         ↓                 │
State → LLM → Tool Call ──┘ → Output
```

**LangGraph 的核心概念：**

```python
from langgraph.graph import StateGraph, MessageGraph

# 定义一个状态图
graph = StateGraph(AgentState)

# 添加节点
graph.add_node("agent", call_model)
graph.add_node("tools", tool_executor)

# 添加条件边——这是 Graph 比 Chain 强大的关键
graph.add_conditional_edges(
    "agent",
    should_continue,  # 决策函数：继续循环还是结束？
    {
        "continue": "tools",
        "end": END
    }
)
graph.add_edge("tools", "agent")

# 编译
app = graph.compile()
```

**LangGraph 的优势：**

- **状态持久化**：Agent 可以在多轮交互中保持状态
- **循环支持**：ReAct 循环是原生支持的，不需要 hack
- **人机协同**：可以在循环中插入人工审核节点（Human-in-the-loop）
- **流式输出**：支持中间步骤的实时查看

### 4. LangSmith——LLM 应用的"可观测性"

LLM 应用有一个独特的调试难题：

```
传统应用：
代码错了 → 抛出异常 → 修复 → 重新部署

LLM 应用：
输出错了 → 是 Prompt 问题？还是检索问题？还是模型问题？
          → 这次错了，下次可能对——是不是随机的？
          → 怎么复现？
```

**LangSmith** 是 LangChain 生态的"监控仪表盘"：

```
LangSmith 能回答的问题：
├── 每次 LLM 调用花了多少 Token、多少钱？
├── 哪个 Prompt 模板的准确率最高？
├── RAG 检索到的文档是否相关？
├── Agent 的每一步在干什么？花了多长时间？
└── 用户满意度如何？（通过反馈评分）
```

**关键能力：**

| 能力 | 说明 | 开发者的收益 |
|:-----|:-----|:------------|
| Trace | 完整记录每次调用的输入/输出/中间步骤 | 调试"黑盒"问题 |
| Evaluation | 用测试集自动评估 Prompt 变体 | 数据驱动优化 |
| Annotation | 人工标记好坏样本 | 构建评估数据集 |
| Monitoring | 生产环境实时监控 | 第一时间发现问题 |
| Hub | Prompt 版本管理 | 团队协作 |

### 5. LanqChain 生态总览与选型建议

**什么时候用 LangChain？**

```
你的应用需要：
├── 多步链式调用 → LangChain Chains
├── RAG 流程 → LangChain Retrieval
├── Agent 循环 → LangGraph
├── 多模型切换 → LangChain Model Abstractions
├── 生产监控 → LangSmith
└── 以上全部 → LangChain 全家桶
```

**什么时候不用 LangChain？**

```
你的场景：
├── 只是一个简单的单轮 Prompt 调用 → 直接用 API
├── 需要极致的性能优化 → 手写（减少框架开销）
├── 边缘设备/嵌入式系统 → llama.cpp 等轻量方案
└── 只是学习原理 → 不要用框架，手写理解后再用
```

**2026 年的框架格局：**

| 框架 | 定位 | 优势场景 | 学习曲线 |
|:-----|:-----|:---------|:---------|
| LangChain | 全功能框架 | 快速原型→生产 | 中等 |
| LlamaIndex | 数据/检索为中心 | RAG 密集型应用 | 低 |
| Haystack | 企业搜索与 NLP | 文档 QA、搜索 | 中等 |
| AutoGen | 多 Agent 对话 | 多 Agent 协作 | 高 |
| CrewAI | 多 Agent 角色扮演 | 团队式 Agent 任务 | 低 |
| **手写代码** | 零依赖 | 学习/极致优化 | 高（自由度也高） |
