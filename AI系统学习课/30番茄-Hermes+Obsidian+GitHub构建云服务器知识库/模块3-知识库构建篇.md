## 🍅 番茄12：LLM-WIKI知识库架构

> ⏱ 25分钟 | 📌 核心技能：AI友好的知识库结构设计

### 12.1 什么是LLM-WIKI？

LLM-WIKI是一种专为大语言模型优化的知识库组织模式：

```
传统知识库 → 分层文件夹（人类友好）
LLM-WIKI   → 密集链接网络（AI友好 + 人类友好）
```

### 12.2 LLM-WIKI核心原则

```
原则1：前端加载（Front-loading）
  → 最重要的信息放在笔记开头
  → AI阅读时，前100字决定理解质量
  
原则2：密集链接（Dense Linking）
  → 每个概念至少链接到3-5个相关概念
  → AI通过链接关系理解知识网络
  
原则3：元数据驱动（Metadata-driven）
  → YAML frontmatter 提供结构化信息
  → AI可快速过滤和检索
  
原则4：原子化（Atomicity）
  → 每个笔记只讲一个概念
  → 概念既不过大也不过小
```

### 14.2 配置自动同步

#### 方案一：Cron定时同步（推荐）

```bash
# 创建同步脚本
nano ~/sync-knowledge.sh
```

```bash
#!/bin/bash
# ~/sync-knowledge.sh - 知识库自动同步脚本

REPO_PATH="$HOME/knowledge-bases/main"
LOG_FILE="$HOME/logs/sync-knowledge.log"

echo "[$(date)] 开始同步..." >> "$LOG_FILE"

cd "$REPO_PATH" || {
    echo "[ERROR] 无法进入仓库目录" >> "$LOG_FILE"
    exit 1
}

# 拉取最新更改
git pull origin main 2>&1 >> "$LOG_FILE"

# 记录同步状态
if [ $? -eq 0 ]; then
    echo "[SUCCESS] 同步完成" >> "$LOG_FILE"
else
    echo "[ERROR] 同步失败" >> "$LOG_FILE"
fi

echo "---" >> "$LOG_FILE"
```

```bash
# 赋予执行权限
chmod +x ~/sync-knowledge.sh

# 创建日志目录
mkdir -p ~/logs

# 配置cron定时任务（每30分钟同步一次）
crontab -e
```

添加以下行：
```cron
# 每30分钟同步知识库
*/30 * * * * /home/hermes/sync-knowledge.sh

# 每天凌晨3点强制同步
0 3 * * * cd /home/hermes/knowledge-bases/main && git fetch origin && git reset --hard origin/main >> /home/hermes/logs/force-sync.log 2>&1
```

#### 方案二：Git Webhook自动同步（高级）

当GitHub仓库有更新时，自动触发服务器同步。

1. 在服务器上创建Webhook监听端：

```bash
# 创建Webhook脚本
nano ~/webhook-server.py
```

```python
#!/usr/bin/env python3
"""简单的GitHub Webhook服务器"""
import subprocess
import sys
from http.server import HTTPServer, BaseHTTPRequestHandler

class WebhookHandler(BaseHTTPRequestHandler):
    def do_POST(self):
        content_length = int(self.headers['Content-Length'])
        self.rfile.read(content_length)
        
        # 执行同步脚本
        result = subprocess.run(
            ['/home/hermes/sync-knowledge.sh'],
            capture_output=True, text=True
        )
        
        self.send_response(200)
        self.end_headers()
        self.wfile.write(b'OK')
        
        print(f"[Webhook] 同步完成: {result.stdout}")

if __name__ == '__main__':
    server = HTTPServer(('0.0.0.0', 9000), WebhookHandler)
    print('Webhook server running on port 9000...')
    server.serve_forever()
```

```bash
# 安装依赖
sudo apt install -y python3

# 运行webhook服务（使用screen或systemd）
screen -S webhook
python3 ~/webhook-server.py
```

2. 在GitHub仓库设置Webhook：
   - GitHub → 仓库 → Settings → Webhooks → Add webhook
   - **Payload URL**: `http://114.215.208.216:9000`
   - **Content type**: `application/json`
   - **Secret**: 可选，增加安全性

## 🍅 番茄15：本地-服务器-GitHub三方同步

> ⏱ 25分钟 | 📌 核心技能：三端同步架构、冲突解决方案、多设备工作流

### 15.1 三端同步架构

```
         ┌──────────────────┐
         │   本地电脑 Obsidian │
         │   (你写笔记的地方)  │
         └────────┬─────────┘
                  │ git push / pull
                  ▼
         ┌──────────────────┐
         │    GitHub远程仓库  │ ←── 中心枢纽
         └────────┬─────────┘
                  │ git pull / push
                  ▼
         ┌──────────────────┐
         │  阿里云服务器Hermes │
         │   (AI Agent运行)   │
         └──────────────────┘
```

### 15.2 多设备工作流

**场景1：你只有一台电脑 + 服务器**
```
写笔记 → 自动提交 → push到GitHub → 服务器自动pull → Hermes读取
```
简单模式，无需任何手动操作。

**场景2：多台电脑（公司+家里）**
```
公司电脑写笔记 → 自动提交 → push到GitHub
                                    ↘
                              GitHub最新版本
                                    ↗
家里电脑 ← 自动pull ← ← ← ← ← ← ←
```
最常用模式，两台电脑都配置Obsidian Git自动同步。

**场景3：手机上阅读笔记**
```
通过GitHub mobile APP查看笔记
或使用Obsidian的第三方同步插件读取GitHub仓库
```

### 15.4 冲突预防与处理

**预防措施**：
1. 不同设备不要同时编辑同一文件
2. 重要编辑先手动pull再修改
3. 使用Obsidian Git的"拉取前提交更改"功能

**冲突处理**：

当出现`<<<<<<< HEAD`标记时：

```markdown
<<<<<<< HEAD
这是服务器上的版本
=======
这是你刚修改的版本
>>>>>>> main

# 手动选择保留哪个版本
# 删除 <<<<<<<、=======、>>>>>>> 标记
```

**快速解决**：

```bash
# Obsidian命令面板
Ctrl+Shift+P → "Obsidian Git: Pull" 
# 如果有冲突，Obsidian会提示，手动编辑解决

# 或者通过终端强制覆盖（谨慎）
git fetch origin
git reset --hard origin/main  # 这会丢弃本地所有更改！
```

name: "知识库助手"

system_prompt: |
  你是一个知识库助手，基于用户的Obsidian知识库回答问题。
  
  核心原则：
  1. 优先从知识库中查找相关信息，而不是依赖自己的训练数据
  2. 如果知识库中有相关信息，引用具体来源
  3. 如果知识库中找不到相关信息，诚实告知并尝试提供一般性帮助
  4. 回答要结构化，清晰易懂
  5. 当涉及专业领域（心理/法律/写作）时，明确说明这是知识库中的知识
  
  回答格式：
  - 先给出直接答案
  - 然后引用知识库来源（[[笔记名称]]）
  - 最后提供延伸阅读建议

knowledge_base:
  enabled: true
  base_path: "/home/hermes/knowledge-bases/main"
  search_top_k: 5  # 检索top-5相关文档
  
tools:
  - web_search      # 允许联网搜索补充
  - calculator      # 允许计算
