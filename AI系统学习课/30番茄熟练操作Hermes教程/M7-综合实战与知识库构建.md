## 🍅 番茄25：整体架构设计（25分钟）

### 25.1 为什么需要知识库？

Hermes 虽然有三层记忆系统，但它的记忆容量有限（max_tokens 限制），而且 MEMORY.md 不适合存放大量结构化知识。

**解决方案**：构建外部知识库（LLM-WIKI 架构），让 Hermes 通过 RAG 检索知识。

### 25.2 LLM-WIKI 架构

> 本架构从 [[Claude_Memory/2026-06-20-Hermes-教程创作洞见]] 提炼。

**传统知识库的问题：**
- 文件夹层级过深 → AI 检索时丢失上下文
- 缺少元数据 → AI 无法判断信息可信度
- 孤立笔记 → AI 无法建立知识关联

**LLM-WIKI 的解决方案：**
- ✅ YAML frontmatter 提供结构化元数据
- ✅ 密集双向链接形成知识网络
- ✅ "一句话定义"帮助 AI 快速理解
- ✅ "费曼解释"提供通俗类比

### 25.3 三端同步架构

```
┌─────────────────────────────────────────────────────────────────┐
│                   三端同步架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│                                                                 │
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐  │
│  │  写笔记       │ ──►  │  GitHub 仓库  │ ◄──  │  Hermes 读取  │  │

│  │  标签管理     │      │  版本控制     │      │  Agent 回答  │  │
│  └──────────────┘      └──────────────┘      └──────────────┘  │
│        │                      │                      │          │
│        │  Obsidian Git        │  15min 自动同步       │  Cron    │
│        │  插件自动提交        │                      │  30min拉 │
│        ▼                      ▼                      ▼          │
│  自动备份 ✓              中央存储 ✓              实时检索 ✓     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 25.4 知识库中的条目模板

每个知识条目遵循统一格式：

```markdown
---
title: 概念名称
type: concept
tags: [分类标签]
created: 2026-06-22
source: 来源
confidence: high  # high/medium/low
---

# 概念名称

**一句话定义**：[用一句通俗的话说清楚]

## 详细说明
[2-3段详细解释]

## 费曼解释
[用大白话/类比/故事来解释，让非专业人士也能理解]

## 关键要点
- 要点1
- 要点2

## 关联概念
- [[相关概念1]]
- [[相关概念2]]

## 来源
- [来源链接或参考文献]
```

## 🍅 番茄26：在服务器上配置知识库（25分钟）

### 26.1 在 Obsidian 中创建知识库

首先在本地创建结构化知识条目。以"心理学知识库"为例：

```markdown
---
title: 认知失调
type: concept
tags: [心理学, 社会心理学]
created: 2026-06-22
confidence: high
---

# 认知失调

**一句话定义**：当一个人的行为与信念不一致时，产生的心理不适感。

## 费曼解释
就像你明明知道吸烟有害健康，却还在抽烟。
你的大脑为了减轻这种矛盾感，会找各种理由安慰自己
（"就抽一根没事的"、"xxx抽了一辈子也活到90岁"）。
这个自我说服的过程就是减少认知失调。

## 关键要点
- 由 Leon Festinger 在 1957 年提出
- 失调程度取决于不一致的重要性和数量
- 减少失调的三种方式：改变行为、改变信念、寻找新理由

## 关联概念
- [[自我合理化]]
- [[确认偏误]]
```

### 26.2 将知识库推送到 GitHub

```bash
# 在 Obsidian 知识库目录中初始化 Git
cd /path/to/your/vault
git init
git add .
git commit -m "初始化知识库"

# 连接 GitHub 远程仓库
git remote add origin https://github.com/你的用户名/你的仓库.git
git branch -M main
git push -u origin main
```

### 26.3 在服务器上克隆知识库

```bash
# SSH 登录到你的云服务器
ssh ubuntu@你的服务器IP

# 克隆知识库
git clone https://github.com/你的用户名/你的仓库.git ~/knowledge-base

# 设置定时同步（每30分钟拉取最新内容）
crontab -e
# 添加以下行：
*/30 * * * * cd ~/knowledge-base && git pull origin main 2>&1 >> ~/kb-sync.log
```

> ⚠️ **冲突预防**：使用"本地为主、服务器只读"策略——只在本地编辑，服务器只拉取。这几乎不会产生 Git 冲突。

### 26.4 避免 Git 冲突的最佳实践

```bash
# 如果服务器上不小心有了本地修改，强制覆盖：
cd ~/knowledge-base
git fetch origin
git reset --hard origin/main
```

## 🍅 番茄27：配置 Hermes RAG 检索（25分钟）

### 27.1 让 Hermes 读取知识库

通过 MEMORY.md 告诉 Hermes 知识库的位置和用途：

```markdown
# Hermes 长期记忆

## 知识库信息
- 知识库位置：~/knowledge-base/
- 知识库结构：按学科分类的 LLM-WIKI 格式条目
- 支持格式：Markdown + YAML frontmatter
- 更新频率：每30分钟自动同步

## 使用规则
1. 回答问题时优先从知识库检索
2. 引用知识库内容时标注来源
3. 如果知识库中找不到答案，再使用模型自身知识
```

### 27.2 测试知识库检索

```bash
hermes chat
```

```
You: 请从我的知识库中查找关于认知失调的资料。

Hermes: 根据你的心理学知识库，我找到了以下信息：

**认知失调**（来自知识库）
- 定义：当一个人的行为与信念不一致时产生的心理不适感
- 提出者：Leon Festinger（1957年）
- 费曼解释：就像明知吸烟有害却还在抽，大脑找各种理由安慰自己...
- 关联概念：自我合理化、确认偏误
```

### 27.3 创建"知识库查询"Skill（可选）

为了让查询更方便，可以创建一个专门的 Skill：

```markdown
# 技能名称: 知识库查询

## 描述
从本地知识库中检索和查询信息

## 触发词
- 查一下
- 知识库
- 搜索知识

## 执行流程
1. 解析用户查询的关键词
2. 在 ~/knowledge-base/ 中搜索匹配文件
3. 读取匹配文件的内容
4. 按 LLM-WIKI 格式提取关键信息
5. 返回结构化回答

## 参数
- 关键词: 用户输入的搜索内容
- 范围: 全部（可指定学科）
```

```bash
# 将 Skill 保存到 skills/custom/
nano ~/.hermes/skills/custom/knowledge-query.md
```

### 27.4 知识库质量 vs 数量的原则

```
1个高质量条目 > 10个速成条目

每个条目必须包含：
✅ YAML 元数据（标题、标签、置信度）
✅ 一句话定义（帮助 AI 快速理解）
✅ 费曼解释（通俗类比）

经验值：
- 心理学：30个条目覆盖80%常见问题
- 写作：40个条目覆盖主要创作理论
- 法律：50个条目覆盖常见民事纠纷
```
