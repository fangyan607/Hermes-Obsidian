## 二、架构决策记录

### 2.1 为什么选用 LLM-Wiki + GBrain 双系统？

| 维度 | LLM-Wiki | GBrain |
|------|----------|--------|
| 优势 | 文件级管理，批量操作高效，AI 检索友好 | 概念级关联，语义关系清晰，可视化直观 |
| 劣势 | 实体关系隐式（依赖 tags/wikilinks） | 维护成本高，需手工设计关系 |
| 适用场景 | 大量文档的知识库（书籍、法条） | 少量核心概念的图谱化 |

**决策**：双系统并行，LLM-Wiki 做主体存储，GBrain 做知识索引。

### 2.2 目录结构设计原则

```
LLM-Wiki/{domain}/
├── .书库索引/    # 隐藏目录，存放元数据和管理文档
├── schema/       # LLM-Wiki 插件配置
├── Log.md        # 操作日志
└── ...书籍文件
```

- `.书库索引/` 使用隐藏目录前缀 `.` 避免与书籍文件混淆
- `schema/` 为 LLM-Wiki 插件标准配置目录
- 书籍文件直接放在领域根目录（或按子目录分类）

### 2.3 YAML Frontmatter 字段设计

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | 是 | 文件名（自动生成） |
| `category` | 是 | 领域分类（自动匹配目录名） |
| `type` | 是 | 领域标识（小写英文） |
| `skill_tag` | 增强 | 关联 SKILL 标签 |
| `level` | 增强 | 入门/进阶/专业 |
| `keywords` | 增强 | 核心关键词 |

### 2.4 GBrain 实体设计原则

- **粒度控制**：每个实体代表一个可独立讨论的概念/理论/方法
- **关系密度**：每个实体至少连接 2 个其他实体
- **跨域桥接**：每个领域至少建立 2 条跨域连接
- **类型规范**：law/theory/practice/method/concept/therapy 六种实体类型

### 2.5 批处理策略

- 使用原子操作 `{ echo frontmatter; cat file; } > tmp && mv tmp file` 避免文件损坏
- 跳过已有 frontmatter 的文件（`head -1` 检测 `---`）
- 跳过 `Log.md` 和 `Obsidian-LLMWIKI*` 系统文件
- 按子目录分类处理法律库（中国法律/行政法规/司法解释/法律相关/年度案例）

## 四、经验教训

### 4.1 技术问题

| 问题 | 原因 | 解决 |
|------|------|------|
| Bash 沙箱阻止绝对路径 glob | `/*/` 被解释为根目录路径 | `cd` 进入目标目录后再用相对路径 |
| Bash 沙箱阻止 `2>/dev/null` | 沙箱安全策略 | 移除 stderr 重定向，改用条件判断 |
| 任务 ID 过期不可查 | agent 长时间运行后 task ID 失效 | 通过检查文件系统状态验证完成情况 |
| 学习方法的索引只列了 10 本书 | 遗漏了 5 本后添加的书籍 | 手动补充更新 |

### 4.2 流程改进建议

1. **先评估再动手**：在创建前先全面了解资料数量和分布，避免遗漏
2. **批处理脚本先行**：大量文件操作前先在小样本上测试脚本
3. **分步验证**：每完成一步都做局部验证，而不是等全部完成才检查
4. **GBrain 提前设计关系图**：在创建实体页面前先画好关系图谱，确保连接完整
5. **统一 type 规范**：新领域创建时先确定 type 值，避免后续不一致
