## 🔄 第三部分：Hermes 原生双向同步方案（要求2）

### 3.1 现有方案的问题

现有 `Obsidian Git → GitHub → Cron pull` 是**单向同步**——只能本地写、服务器读。Hermes 不能把学习到的知识"写回"知识库。

### 3.2 双向同步架构

```
        写笔记                        自动提交 + 推送
本地 Obsidian ─────────────→ Obsidian Git ─────────────→ GitHub
    ↑                                                       │
    │                                                       │  cron pull
    │                                                       ▼
    │                                              腾讯云服务器 Hermes
    │                                                       │
    │                                                       │ Hermes Knowledge
    │                                                       │ Skill 自动写入
    │                                                       ▼
    └──────────── git pull ──────────── GitHub ←──── Hermes 生成新条目
```

### 3.3 Hermes 写回机制的三种实现

#### 方案A：Skill 自动生成写回（推荐）

配置 Auto-Generate Skill 将对话中的知识写回知识库：

```yaml
# config.yaml
skills:
  enabled: true
  auto_generate: true      # 开启自动生成
  knowledge_writeback:     # 写回配置
    enabled: true
    target_dir: "/home/ubuntu/knowledge-bases/main/书库/"
    format: llm-wiki       # 使用 LLM-WIKI 格式
    commit_on_write: true  # 写回后自动 git commit
```

创建一个"知识写回"Skill：

```markdown
# 技能名称: 知识库写入

## 描述
将从对话中学习到的新知识以 LLM-WIKI 格式写回知识库

## 触发词
- 记住这个
- 写入知识库
- 新建条目
- 记下来

## 执行流程
1. 解析用户提供的新知识
2. 识别知识所属领域（心理学/中医学/法律）
3. 生成 LLM-WIKI 格式条目（含 YAML frontmatter）
4. 写入对应领域目录
5. 在 Git 中提交更改
6. 确认写入成功

## 参数
- 领域: 自动识别（可手动指定）
- 置信度: 从对话中推断
- 关联概念: 自动匹配已有条目
```

#### 方案B：系统提示词写回（轻量）

在 MEMORY.md 中配置写回指令：

```markdown
# Hermes 长期记忆

## 知识库写回规则
当你学到新知识或确认某个知识点需要补充时：
1. 生成 LLM-WIKI 格式条目
2. 写入 ~/knowledge-bases/main/对应领域/
3. 执行: cd ~/knowledge-bases/main && git add . && git commit -m "🤖 Hermes 自动更新: [主题]"
4. 告知用户已更新知识库

## 写回格式
使用以下 YAML frontmatter + 正文结构：
---
title: 主题
type: concept
domain: 心理学|中医学|法律
confidence: high|medium|low
---
```

#### 方案C：定时自动整理（高级）

配置 Cron 脚本，让 Hermes 定期整理和补充知识库：

```bash
# /home/ubuntu/knowledge-sync.sh
#!/bin/bash

# 1. 拉取最新知识库
cd /home/ubuntu/knowledge-bases/main
git pull origin main

# 2. 执行知识库健康检查
# （Hermes 自动分析知识库中的空白/待完善条目）

# 3. 提交任何 Hermes 写入的更改
cd /home/ubuntu/knowledge-bases/main
git add .
git diff --quiet || git commit -m "🔄 定时同步: $(date +%Y-%m-%d)"
git push origin main
```

### 3.4 完整双向同步流程

```
┌─────────────────────────────────────────────────────────────────┐
│              双向同步完整工作流                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
```

## ✅ 第六部分：审定结论

### 方案修改总结

| 修改项 | 原方案 | 审定后 |
|:-------|:-------|:--------|
| 知识库方法 | LLM-WIKI 单一 | LLM-WIKI 主力 + RAG 书库增强 |
| 同步方向 | Obsidian→GitHub→服务器（单向） | Obsidian ↔ 服务器（双向，Hermes可写回） |
| 知识域 | 心理+法律+写作 | 心理+中医学+法律（替换写作为中医） |
| 书库利用 | 未涉及 | 直接映射103册中医书库+106卷道医 |
| Agent 管理 | 手动编辑服务器 YAML | Agent配置纳入Git版本控制 |
| TCM 支持 | 无 | 新增中医知识库+中医Agent+书库RAG |

### 关键决策点

1. **为什么替换写作为中医？** 书库中已有103册中医书库+106卷道医全书——这是已经存在的资产，写作域的内容相对较少。修改后三域覆盖：心理（人文）+ 中医（传统医学）+ 法律（社会科学），形成更有价值的知识体系。

2. **为什么 LLM-WIKI 为主而非 RAG？** Hermes 的 MEMORY.md 原生支持 LLM-WIKI 模式，集成成本最低。RAG 仅作为中医海量书库的补充增强层。

3. **为什么双向同步可行？** Hermes 的 Skill 自动生成机制已经具备了"写"的能力——只需将 auto_generate 的目标目录指向知识库，再配合 git commit 即可完成闭环。
