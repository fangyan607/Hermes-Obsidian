# Day16：Prompt Engineering——与AI对话的艺术（番茄76-80）

> **Part 4 · 大语言模型与智能体**
> **悬疑探案进度**：你学会了LLM的"大脑结构"，现在该学"如何给它下指令"了。
> **番茄范围**：🍅76-80

---

## 悬疑开场：同一个AI，同一个问题，完全不同的答案

**实验**：

> 问题1："给我写个Python函数。"

AI回复：`def my_function(): pass`（敷衍）

> 问题2："你是一位资深Python工程师。你的任务是：编写一个函数，接收一个URL列表，返回其中所有可访问的URL。要求：① 使用异步HTTP请求提高效率 ② 设置3秒超时 ③ 优雅处理所有异常。输出格式：完整代码 + 使用示例 + 注意事项。"

AI回复：（给你写了50行高质量代码）

---

**同一个AI。同一个底层模型。只是prompt不同。**

差距不是模型决定的——是**你**决定的。

这就是Prompt Engineering的底层逻辑：**你不是在"提问"，你是在"编程"**——用自然语言编程。

## 🍅 番茄76 · T1：Prompt的本质——不是"提问"，是"编程"

### 🔍 犯罪现场：一个被严重低估的事实

大多数人以为和AI对话就像和人对话。错了。

**和人对话**：你说"帮我弄一下这个"，对方理解你的意图，自己补全缺失的信息。

**和AI对话**：你说什么，它听什么。**你不写清楚，它不猜**（或者猜错）。

### 🧠 核心洞察：AI的"心理模型"

要写好Prompt，你必须理解AI的"心理模型"——它怎么"看待"你的问题：

| 人类期望 | AI的实际行为 |
|:---------|:------------|
| "你懂我意思吧？" | 不，我不懂，请写清楚 |
| "省略细节你应该也知道" | 不，我不知道你省略了什么 |
| "我随便说说，你认真点" | 你随便说，我也随便答 |
| "这里应该理解为... " | 你的话有歧义，我随机选一种 |

**Prompt的本质不是"提问"——Prompt是在实例化一个"虚拟程序员"。**

### 🧠 Prompt Engineering的核心三问

每次写Prompt之前，问自己三个问题：

| 问题 | 类比编程 | 具体操作 |
|:-----|:---------|:---------|
| ① **我要什么角色？** | 选API实例 | "你是一个...[角色设定]" |
| ② **我要什么输出？** | 定义返回值类型 | "输出格式：JSON/Markdown/代码" |
| ③ **我有什么约束？** | 设置参数 | "不能...必须...不超过..." |

> ✋ **费曼自测**：为什么说"Prompt不是提问，是编程"？请用"雇一个临时工"做类比来解释。

### 🧠 一个形象的对比

```
差Prompt（就像说"给我做饭"）：
  你：写一个网站
  AI：<写了一个很基础的HTML页面>

好Prompt（就像说"做一桌川菜，要麻婆豆腐、水煮鱼、回锅肉……"）：
  你：你是一位全栈工程师。请为一个宠物商店开发一个响应式网站首页。
  要求：
  - 使用React + Tailwind CSS
  - 包含导航栏、商品展示区、购物车预览
  - 移动端优先设计
  - 代码完整可运行
  AI：<写出完整项目结构>
```

**差距不在AI，在prompt里包含的信息量。**

## 🍅 番茄77 · T2：结构化Prompt——角色、任务、格式、约束、示例

### 🔍 犯罪现场：把模糊的请求变成精确的指令

AI不是读心术专家。你需要把你的意图翻译成它"听得懂"的结构。

### 🧠 结构化Prompt五大要素

```
┌─────────────────────────────────────────────────────────┐
│                结构化Prompt 五要素                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [角色] "你是一个资深Python开发者"                       │
│     ↓ 设定身份、知识范围、语气风格                         │
│                                                         │
│  [任务] "请编写一个异步爬虫"                              │
│     ↓ 明确"做什么"——越具体越好                            │
│                                                         │
│  [格式] "输出为Markdown表格+代码块"                      │
│     ↓ 告诉AI"怎么做输出"——控制结构                         │
│                                                         │
│  [约束] "不要用第三方库，Python 3.11+语法"                │
│     ↓ 限制边界——避免AI走偏                                 │
│                                                         │
│  [示例] "输入：['url1','url2'] → 输出：['ok','timeout']"  │
│     ↓ Few-shot——给一个"正确答案"模板                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 🧠 五要素实战拆解

#### 1️⃣ 角色设定（System Prompt / Persona）

```
❌ 差："帮我改一下这段代码"
✅ 好："你是一位代码审查专家，专精于Python性能优化和安全审计。
        请以严格的标准review以下代码："
```

**为什么角色设定有用？** 因为训练数据中，不同"角色"的文本风格不同。设定角色等于激活了训练数据中对应"角色"的知识区域。

#### 2️⃣ 任务描述（Task）

```
❌ 差："解释量子计算"
✅ 好："用500字以内，向一个高中生解释量子计算的核心概念。
        重点在"量子比特"和"叠加态"，避免数学公式。
        用日常生活中的类比来说明。"
```

**任务越具体，AI的输出越可控。**

#### 3️⃣ 输出格式（Format）

```
❌ 差："列出TODO list"
✅ 好："请按以下格式输出：
         | 优先级 | 任务名 | 预估时间 | 依赖 |
         | P0     | 登录页 | 2天      | 无   |
         | P1     | 支付   | 3天      | 登录 |
```

#### 4️⃣ 约束条件（Constraints）

```
❌ 差："写个广告文案"
✅ 好："写一个200字以内的广告文案，针对25-35岁职场人士。
        禁止使用'最'、'第一'、'极致'等极限词汇。
        语气：专业、轻松、带一点幽默。"
```

#### 5️⃣ 示例（Examples / Few-shot）

这是最强大的技术之一：

```
❌ 差："把这段话改得更有说服力"
✅ 好："请按以下示例进行改写：

输入：'这个产品不错。'
输出：'为什么超过10,000名用户选择了这个产品？三个理由让你无法拒绝。'

现在请改写：'这个软件很好用。'"
```

> ✋ **费曼自测**：为什么"给示例"比"说要求"更有效？请从"人类学习"的角度解释。

### 🧠 一个完整的结构化Prompt模板

```
# 角色
你是一位[领域]专家，擅长[技能]。

# 任务
请[具体任务描述]。

# 格式要求
- 输出格式：[Markdown/JSON/代码/表格]
- 长度：[字数/段落数]
- 结构：[分段方式]

# 约束条件
- ✅ [必须包含的内容]
- ❌ [禁止包含的内容]
- ⚠️ [需要注意的事项]

# 示例
输入：[示例输入]
输出：[示例输出]

# 当前输入
[实际要处理的内容]
```

## 🍅 番茄78 · T3：高级技巧——Chain-of-Thought、Few-shot、Self-Consistency、ReAct

### 🔍 犯罪现场：从"直接回答"到"边推理边回答"

有时候，直接问AI它答错；但如果你让它"先推理再回答"，它突然就对了。

这就是**高级Prompt技巧**的威力。

### 🧠 技巧一：Chain-of-Thought（CoT，思维链）

**核心思想**：让AI在给出最终答案前，先展示推理过程。

```
❌ 直接问：
  Q：一个球和一个球拍共1.10美元，球拍比球贵1美元，球多少钱？
  A：0.10美元 ❌（大多数人直觉错误，AI也一样）

✅ 思维链：
  Q：一个球和一个球拍共1.10美元，球拍比球贵1美元，球多少钱？
  A：设球的价格为x，则球拍为x+1。
      总价：x + (x+1) = 1.10
      2x + 1 = 1.10
      2x = 0.10
      x = 0.05
      所以球是0.05美元 ✅
```

**为什么CoT有效？** 因为AI的"直觉回答"往往和人类一样有偏差。强迫它"展示推理"会激活更深层的逻辑处理能力。

### 🧠 技巧二：Few-shot Learning（少样本学习）

**核心思想**：在prompt里给2-3个完整示例，让AI"学会"模式。

```
请将以下句子翻译为"海盗语"：

示例1：
  正常：Hello, how are you?
  海盗：Ahoy, how be ye?

示例2：
  正常：I'm going to the store.
  海盗：I'm headin' to the treasure chest.

示例3：
  正常：See you later!
  海盗：Fair winds to ye!

现在翻译：
  正常：Can you help me?
```

**Few-shot vs Zero-shot 的效果差距，在复杂任务上可能是天壤之别。**

### 🧠 技巧三：Self-Consistency（自一致性）

**核心思想**：同一个问题问多次，取"最一致"的答案。

```
步骤：
1. 同一个prompt，调用AI多次（建议3-5次）
2. 收集所有回答
3. 选出现频率最高的答案（多数投票）

原理：随机性会导致不同的推理路径，
      但正确的答案往往在多条路径中都出现。
```

**这就像**：让5个侦探独立破案，然后对比他们的结论。如果5个人都得出了同样的结论，那大概率是对的。

### 🧠 技巧四：ReAct（Reason + Act）

**核心思想**：让AI在"推理"和"行动"之间交替，每次行动后观察结果再推理。

这个技巧太重要了，Day18会专门讲Agent架构时深入拆解。先留个悬念。

> ✋ **费曼自测**：比较CoT和Few-shot——它们分别解决什么问题？在什么场景下该用哪个？

### 🧠 技巧速查表

| 技巧 | 一句话 | 什么时候用 | 复杂度 |
|:-----|:-------|:-----------|:------:|
| **CoT** | "先想后答" | 推理/数学/逻辑问题 | ⭐⭐ |
| **Few-shot** | "给例子" | 输出格式复杂/需要模仿风格 | ⭐ |
| **Self-Consistency** | "多问几次取一致" | 高可靠性要求的场景 | ⭐⭐⭐ |
| **ReAct** | "边想边做" | 多步任务（Agent场景）| ⭐⭐⭐⭐ |
| **Role Prompting** | "扮演专家" | 需要特定领域知识 | ⭐ |

## 🍅 番茄79 · T4：Prompt模式库——从总结到创作到分析的模板

### 🔍 犯罪现场：不是每次都要重新发明轮子

好的Prompt是有模式的。掌握10个核心模板，覆盖80%的日常需求。

### 🧠 一、总结类

```
你是一位资深编辑。请总结以下文本的核心观点：
- 控制在200字以内
- 使用项目符号列出要点
- 保留原文的关键数据和引用

文本：
[粘贴内容]
```

### 🧠 二、创作类

```
你是一位创意写作教练。请帮我写一篇[文章类型]：

主题：[主题]
目标读者：[读者画像]
语气：[正式/轻松/幽默/严肃]
长度：[字数]
结构：
1. 引人入胜的开头（一个故事/一个问题/一个数据）
2. 核心论点（3-5个段落，每段一个观点+证据）
3. 有力的结尾（总结+行动号召）
```

### 🧠 三、分析类

```
你是一位数据分析师。请对以下内容进行分析：

分析框架：
1. 关键发现（3-5个最重要的信息）
2. 趋势判断（上升/下降/稳定，依据是什么）
3. 异常点（与预期不符的地方，可能的原因）
4. 建议（基于以上分析的 actionable insights）

内容：
[粘贴数据/报告/文本]
```

### 🧠 四、代码开发类

```
你是一位资深[语言]工程师。请实现以下功能：

需求描述：[功能描述]
技术栈：[框架/库/版本]
测试要求：[单元测试/集成测试]
边界情况：[空值/异常/极端输入]
性能要求：[响应时间/资源消耗]

请输出：
1. 完整可运行代码
2. 使用示例
3. 测试用例
4. 已知限制
```

### 🧠 五、角色扮演/辩论类

```
请扮演两位专家进行辩论：

角色A：[观点A的专家身份]
角色B：[观点B的专家身份]

议题：[争议话题]

格式：
A：[A的观点和论据]
B：[B的反驳和论据]
A：[A的回应]
...

要求：双方论点要有理有据，通过辩论展示问题的多面性。
```

### 🧠 六、教学/解释类

```
你是一位擅长用类比教学的老师。

请解释：[复杂概念]

要求：
1. 先用一个日常生活的类比解释（不超过3句话）
2. 再用一层层深入的"剥洋葱"方式展开
3. 每层用一个问题连接："你可能想问……"
4. 最后用一张表格总结核心要点

受众：完全零基础的[目标受众]
```

> ✋ **费曼自测**：从以上模板中选一个不太熟悉的，用自己的话改写成一个针对你工作场景的模板。

### 🧠 自定义Prompt的"进化"策略

一个好的Prompt通常不是一次写好的，而是迭代出来的：

```
V1：写一个Python函数 → 太宽泛
V2：写一个异步爬虫函数，用aiohttp → 好一点，但还是不清楚
V3：[结构化五要素完整版] → 一次性拿到可用代码
```

**每次迭代的关键**：问自己"AI这次输出哪里不对？"然后补上对应的要素。

| 问题 | 缺少什么要素 |
|:-----|:------------|
| 输出格式不对 | 格式定义 |
| 理解错了任务 | 任务描述不够具体 / 缺示例 |
| 太啰嗦/太简单 | 约束条件（字数、复杂度） |
| 用了不该用的库 | 约束条件（技术栈） |
| 语气不对 | 角色设定 |

## 🍅 番茄80 · T5：实战对比——好Prompt vs 差Prompt的10个案例

### 🔍 犯罪现场：对比出真知

同一个需求，不同prompt，AI的回应天差地别。

### 🧠 案例1：写邮件

```
❌ 差："帮我写封邮件"
✅ 好："你是一位商务沟通专家。请帮我写一封跟进邮件。
        场景：昨天面试了候选人张三，我想发邮件表达感谢并告知下一轮安排。
        语气：专业且亲切。长度：不超过150字。
        关键信息：下一轮面试在下周三，会提前两天通知具体时间。"
```

### 🧠 案例2：代码评审

```
❌ 差："review这段代码"
✅ 好："你是一位资深代码审查专家，专精于[语言/框架]。
        请按以下标准review代码：
        - 安全性（是否有注入/泄露风险）
        - 性能（是否有O(n²)以上复杂度的地方）
        - 可维护性（命名、结构、注释）
        - 测试覆盖（是否有明显缺失的场景）
        对每个问题标注严重等级：🔴高危 / 🟡中危 / 🔵建议"

[代码]
```

### 🧠 案例3：知识问答

```
❌ 差："解释Transformer"
✅ 好："请向一个了解RNN但不懂Transformer的开发者解释Transformer。
        重点对比：
        1. RNN的序列处理 vs Transformer的并行处理
        2. 注意力机制如何替代循环
        3. 位置编码为什么必要
        可以用代码片段辅助说明，但不要贴大段论文原文。"
```

### 🧠 案例4-10：速查表

| # | 场景 | 差Prompt | 好Prompt的关键要素 |
|:-:|:-----|:---------|:------------------|
| 4 | 数据提取 | "提取信息" | 给出列名表 + 输出格式JSON + 示例 |
| 5 | 头脑风暴 | "想些点子" | 给边界（"关于X领域"）+ 数量（"10个"）+ 评级维度 |
| 6 | 改写润色 | "改好一点" | 原风格/目标风格 + 保留要素清单 + 禁止改动清单 |
| 7 | 翻译 | "翻译" | 领域术语表 + 语气要求 + 文化适应注释 |
| 8 | 调试 | "为什么报错" | 完整错误栈 + 预期行为 + 已尝试的排查 |
| 9 | SQL生成 | "写个查询" | 表结构DDL + 期望结果示例 + 性能考量 |
| 10 | 架构设计 | "设计个系统" | 约束条件（用户量/延迟/SLA）+ 现有基础设施 |

### 🧠 "万能Prompt"终极模板

当你不确定怎么写prompt时，用这个：

```markdown
你是一位[角色]，专精于[领域]。

### 任务
[具体的任务描述，包括背景和目的]

### 输出要求
- 格式：[Markdown/JSON/等]
- 长度：[字数/段落数]
- 风格：[正式/轻松/技术性]

### 约束
- ✅ [必须包含]
- ❌ [禁止包含]
- ⚠️ [特定注意事项]

### 参考示例
输入：[示例]
输出：[示例]

### 我的输入
[实际要处理的内容]
```

> ✋ **费曼自测**：选一个你实际工作中会用到的场景，用上面的"万能模板"写出一个prompt。然后发给AI，看结果——然后迭代一次。

### 📌 连线思考

Prompt Engineering的能力不是"记住了多少模板"。它是**结构化思维**的体现——

**你越能清晰地定义角色、任务、格式、约束和示例，你也就越擅长写prompt。**

而这项能力，不仅在和AI对话时有用——在你写技术文档、产品需求、项目管理时同样适用。

### ✅ 费曼三句话（闭眼自测）

1. **Prompt Engineering的本质是什么？** ——你不是在"提问"，你是在"实例化一个虚拟专家"；写prompt就像写代码，定义了角色、任务、格式、约束和示例。
2. **为什么结构化Prompt比随意提问有效？** ——因为AI没有读心术，它只能根据你给的信息做决策；你给的信息越结构化（角色→任务→格式→约束→示例），它的输出就越精准。
3. **高级技巧之间的关系是什么？** ——CoT让AI"先想后答"（解决推理问题），Few-shot让AI"照猫画虎"（解决格式问题），Self-Consistency让AI"多想几次"（解决可靠性问题），ReAct让AI"边想边动"（解决多步任务）。
