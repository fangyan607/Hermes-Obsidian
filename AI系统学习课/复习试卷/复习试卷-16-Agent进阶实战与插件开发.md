### 二、五大 Case 实战拆解

#### Case 1：古诗词绘画（批处理）

| 步骤 | 节点 | 说明 |
|------|------|------|
| Step1 | 工作流 draw_picture | 创建主工作流 |
| Step2 | 大模型节点 | 输入古诗词 → 拆解4个场景描述 |
| Step3 | 代码节点 | 正则提取场景数组(`scenes`, `scene_count`) |
| Step4 | 大模型节点（批处理） | 逐个场景生成文生图提示词 |
| Step5 | text2image插件（批处理） | 逐个提示词生成图片 |
| Step6 | 结束节点 | 输出图片列表 |

**批处理的关键**：当大模型或插件需要对列表中的每个 item 执行相同操作时，开启批处理模式，item1 关联上级输出的 outputList。

**正则提取核心代码**：
```python
scene_pattern = r'场景\d+：([^场景]+)'
for match in re.finditer(scene_pattern, scenes_text, re.DOTALL):
    scenes.append(match.group(1).strip())
```

#### Case 2：智能投顾助手（风险评测与推荐）

**核心流程**：
1. 设计风险评测调查问卷（年龄/投资经验/风险偏好）
2. 创建数据表 `client_risk_assessment`
3. 创建工作流 `product_rec`
4. 开始节点输入：`age, exp_level, risk_tolerance, client_id`
5. 数据库节点：将评测结果保存到数据表
6. 大模型+知识库：根据评测结果匹配产品
7. 知识库：ABC证券公司产品介绍（txt格式，###产品分段）
8. 结束节点：Markdown流式输出
9. 用户界面：表单（收集信息）→ 提交按钮 → 触发工作流 → 结果页面

**筛选规则**：R1/R2/R3 对应 A/B/C 风险等级，新手(exp_level=1)避免推荐复杂衍生品。

#### Case 3：客户分层营销助手

**数据源**：

#### Case 4：智能客服Agent（多Agents模式）

**架构**：
```
用户提问
    │
分诊台（判断问题类型）
    ├── 营销专员 ← 证券知识库
    └── 投诉专员 ← user_tag + user_behavior_event + user_complain
```

**各Agent职责**：
| Agent | 提示词重点 | 技能 |
|-------|-----------|------|
| 分诊台 | 只转发不回复，判断营销or投诉 | 问题分类 |
| 营销专员 | 产品咨询、交易规则解答 | 证券知识库 |
| 投诉专员 | 共情安抚→核实行为→记录投诉 | 数据表查询 |

**投诉流程**：共情→核查用户行为(event_time格式YYYY/M/D H:MM:SS)→确认问题→写入user_complain表。

#### Case 5：市场舆情监测Agent（多工作流嵌套）

**工作流嵌套结构**：
```
工作流 Securities（主流程）
    ├── 插件 securities_past → 新浪财经新闻抓取
    ├── 代码节点 → 日期筛选（same_month_day标志）
    ├── 大模型 → 新闻摘要
    ├── 代码_1 → 过滤空白数据
    ├── 大模型_2 → 热词/风险词提取
    ├── 大模型_1 → 去除"证券""券商""总裁"等字眼
    ├── WordCloud_generation插件 → 词云图
    ├── 工作流 AppStoreEstimate（子流程）→ App评论抓取分析
    │       ├── 插件 AppStorePast → 苹果App Store RSS评论
    │       ├── 大模型（批处理）→ 好评/中性/差评分类
    │       ├── 代码_1 → 好评/差评分离
    │       └── 大模型+词云 → 好评词云/差评词云
    └── 工作流 GenerateDailyReports（子流程）→ 日报生成
            ├── 多个大模型节点 → 分别处理热点/评论/好评/差评
            ├── 文本处理 → 字符串拼接
            └── create_document插件 → PDF导出
```

**舆情监测Agent调度**：人设调用 worklow Securities → 按序执行 → 变量保存中间结果（HotNews, HotWordCloud, GoodWordCloud, BadWordCloud 等）。

#### 问答题

**第1题参考答案：**

**架构：** 用户提问 → 分诊台（单入口调度） → 营销专员（产品咨询） 或 投诉专员（投诉处理）

**分诊台判断规则：**
- 转营销专员：输入涉及产品使用疑问、怎么使用、产品咨询、基础问题时
- 转投诉专员：输入涉及投诉相关、产品使用不成功、用户个人信息、需要后台记录的信息时

**各Agent配置：**
- 营销专员：配置了证券知识库（港股交易规则、上交所规则、平安财富日添利、平安金裕人生等文档）
- 投诉专员：配置了 user_tag（用户标签）、user_behavior_event（用户行为事件）、user_complain（投诉记录表）

**投诉专员处理流程：**
1. 首条回应以共情安抚客户情绪，表达积极解决态度
2. 通过 user_behavior_event 数据表查询并核实用户行为，使用 event_time 做日期判断
3. 如果确实有问题，再写入 user_complain 数据表（含 complain_type：产品使用/系统故障/业务办理）

**第2题参考答案：**

| 维度 | 批处理（古诗词绘画） | 非批处理（智能投顾助手） |
|------|-------------------|---------------------|
| **执行逻辑** | 对列表逐项执行相同操作，一次配置多轮迭代 | 单次输入 → 单次处理 → 单次输出 |
| **适用场景** | 多个同类输入需要相同处理（如4个场景分别生成图片） | 单个输入需要综合推理（如评测结果匹配产品） |
| **节点配置** | 大模型/插件开启批处理模式，item1关联上级outputList | 节点直接关联开始节点或前序节点输出 |

**批处理优势**：同类型任务并行执行，减少重复配置，提高效率。
**非批处理优势**：逻辑简单清晰，适合输入输出1:1的场景。

**市场舆情监测 Agent 中用到批处理的环节：**
- 工作流 Securities 中，对抓取的新闻列表进行日期筛选和摘要提取时使用批处理
- 工作流 AppStoreEstimate 中，对评论列表进行好评/中性/差评分类时使用批处理
