## 🍅 番茄101（T1）：OpenClaw——AI 圈的"隐藏武器"

### 1.1 什么是 OpenClaw？

OpenClaw 是一个**AI 代理编排框架**——简单说，它让不同的 AI 工具和 Agent 能互相配合、协同工作。

```
┌────────────────────────────────────────────────────────────┐
│                      OpenClaw 生态                           │
│                                                              │
│  你的一句指令 → OpenClaw 调度层                              │
│       ↓                                                      │
│   ├─ 理解意图（是什么任务？需要什么工具？）                    │
│   ├─ 分配子任务（拆解为可并行的小任务）                       │
│   ├─ 调度 Agent（谁最适合做这个？Claude？GPT？本地模型？）    │
│   ├─ 编排工具链（需要调用什么 MCP 服务器？）                  │
│   ├─ 监控进度（各子任务完成度、错误处理）                     │
│   └─ 输出结果（汇总、格式化、交付）                           │
└────────────────────────────────────────────────────────────┘
```

**核心哲学**：**"AI Router"——不代替任何 AI，而是让所有 AI 合作。**

### 1.2 OpenClaw 能做什么？

| 能力 | 说人话版 | 真实案例 |
|:-----|:---------|:---------|
| **多模型路由** | 让 Claude 写文章、GPT 做翻译、本地模型处理敏感数据 | 写一篇英文博客，Claude 起草→GPT 润色→本地模型做数据脱敏 |
| **工具编排** | 让 AI 自动调用浏览器、数据库、Slack、邮件 | 客户咨询→读CRM→查库存→生成报价→发邮件 |
| **持久 Agent** | Agent 工作到一半可以暂停，下次继续 | 大数据分析，分多次运行 |
| **人类介入点** | 在关键决策点停下来让你确认 | 自动生成合同→你审核→AI 继续发送 |
| **故障恢复** | Agent 挂了自动重启 | 网络超时→重试→切换备用模型 |

### 1.3 项目示例：用 OpenClaw 构建个人 AI 工作流

**场景**：每天早上，AI 自动准备你的"工作日报"——汇总邮件、Slack 消息、GitHub 活动、日历事件。

**OpenClaw 工作流配置**：

```yaml
# morning-briefing.yaml
name: "每日早报"
trigger: "cron(0 8 * * 1-5)"  # 工作日早上 8 点

steps:
  - agent: "claude-sonnet"
    task: "读取过去24小时的未读邮件，提取重要事项"
    tools: [gmail-mcp]
    
  - agent: "claude-sonnet"  
    task: "读取 Slack 中 @我 的消息和重要频道更新"
    tools: [slack-mcp]
    
  - agent: "claude-sonnet"
    task: "检查 GitHub 通知：我的 PR 有 review 吗？"
    tools: [github-mcp]
    
  - agent: "claude-sonnet"
    task: "查看今日日历，列出所有会议"
    tools: [calendar-mcp]

  - agent: "claude-sonnet"
    task: "汇总以上所有信息，生成一份500字以内的《今日简报》"
    format: "markdown"
    
  - agent: "claude-opus"
    task: "审核简报质量，确认无敏感信息泄露"
    
  - agent: "claude-sonnet"
    task: "将简报发送到我的邮箱和 Slack 私信"
    tools: [gmail-mcp, slack-mcp]
    
human_review: false  # 不需要人工确认
max_retries: 2
```

## 🍅 番茄102（T2）：MCP 协议——Model Context Protocol

### 2.1 什么是 MCP？

**MCP（Model Context Protocol）** 是 Anthropic 在 2024 年底推出的**开放协议**，让 AI 模型能连接任何外部工具和数据源。

**一句话**：**MCP 是 AI 世界的"USB-C 接口"——一个统一标准，连接一切工具。**

```
MCP 之前：                     MCP 之后：

AI → 每个工具一个接口          AI → MCP 协议 → 统一接入层
    ├─ AI → GitHub (自定义API)       ├─ GitHub MCP 服务器
    ├─ AI → Slack (自定义API)        ├─ Slack MCP 服务器
    ├─ AI → 数据库 (自定义API)       ├─ 数据库 MCP 服务器
    ├─ AI → 文件系统 (自定义API)     ├─ 文件系统 MCP 服务器
    └─ AI → ... (N个工具N个接口)     └─ ... (一端开发，多端使用)
```

### 2.2 MCP 的核心架构

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│              │     │                  │     │              │
│  AI 应用     │ ←─→ │   MCP 协议       │ ←─→ │  MCP 服务器  │
│  (Client)    │     │   (标准通信)      │     │  (工具适配器) │
│              │     │                  │     │              │
│  Claude Code │     │  JSON-RPC 2.0    │     │  GitHub MCP  │
│  Cursor      │     │  stdio / HTTP    │     │  Slack MCP   │
│  ChatGPT     │     │  双向通信        │     │  DB MCP      │
│  Codex       │     │                  │     │  Filesystem   │
└──────────────┘     └──────────────────┘     └──────────────┘
```

**三个核心角色**：

| 角色 | 功能 | 例子 |
|:-----|:------|:------|
| **MCP Host** | AI 应用——发起请求的一方 | Claude Code, Cursor, ChatGPT, Codex |
| **MCP Client** | 协议实现——建立与服务器的 1:1 连接 | 内嵌在 Host 中 |
| **MCP Server** | 工具适配——把工具的 API 变成 MCP 标准接口 | `github-mcp`, `filesystem-mcp`, `db-mcp` |

### 2.3 项目示例：配置 GitHub MCP + 文件系统 MCP

#### 配置 GitHub MCP

让 AI Agent 能直接操作 GitHub——读 Issue、提 PR、Review 代码。

```json
// ~/.cursor/mcp.json 或 IDE 的 MCP 配置文件
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_xxxxxxxxxxxxxxxxxxxx"
      }
    }
  }
}
```

**GitHub MCP 能做什么？**

| 操作 | 命令 | AI 自动完成 |
|:-----|:------|:------------|
| 列出 Issue | `list_issues` | "列出所有带有 bug 标签的 open issue" |
| 创建 Issue | `create_issue` | "创建一个新 issue 记录发现的 bug" |
| 读取代码 | `get_contents` | "读取 src/main.ts 的内容" |
| 搜索代码 | `search_code` | "搜索所有地方用到了这个函数" |
| 创建 PR | `create_pull_request` | "创建一个修复 PR" |
| Review PR | `create_review` | "Review 这个 PR，指出所有问题" |

#### 配置文件系统 MCP

让 AI Agent 安全地读写你的文件系统。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", 
               "/Users/you/projects", "/Users/you/documents"],
      "env": {}
    }
  }
}
```

**文件系统 MCP 能做什么？**

| 操作 | 说明 | 安全限制 |
|:-----|:------|:---------|
| `read_file` | 读取文件内容 | 只能在配置的目录范围内 |
| `write_file` | 写入/覆盖文件 | 不能写入系统关键区域 |
| `edit_file` | 编辑文件的部分内容 | 不能执行二进制文件 |
| `list_directory` | 列出目录内容 | 不能访问隐藏目录（除非配置） |
| `search_files` | 搜索文件 | 有文件大小限制 |

## 🍅 番茄103（T3）：n8n——可视化 AI 工作流

### 3.1 什么是 n8n？

n8n 是一个**开源的可视化工作流自动化平台**。你可以把它理解为"AI 版的 IFTTT/Zapier"——但它更强大、更灵活、完全在自己的服务器上运行。

**一句话**：**n8n = 拖拽式编排 + 400+ 集成 + AI 原生支持 + 全本地运行。**

```
传统自动化 vs n8n 自动化：

  Zapier/IFTTT                n8n
  ─────────────────────────────────
  云端运行                    本地运行 or 云端
  封闭生态                    开源（可自托管）
  有限 AI 集成                AI 原生（LLM 节点）
  线性流程                    复杂分支/循环/子工作流
  "如果A就B"                  "如果A就B, 但C时做D, 循环直到E"
```

### 3.2 n8n 的核心概念

| 概念 | 说人话版 | 例子 |
|:-----|:---------|:-----|
| **Trigger（触发器）** | 工作流什么时候启动？ | 收到邮件时 / 每早9点 / Webhook 调用时 |
| **Node（节点）** | 工作流中的每一步动作 | 发送 Slack 消息 / 查询数据库 / 调用 AI |
| **Connection（连接）** | 数据如何在节点间流动 | 邮件内容 → AI 处理 → 写入数据库 |
| **Workflow（工作流）** | 完整的自动化流程 | "收到客户咨询 → AI 生成回复 → 发送 → 记录" |
| **Sub-workflow（子工作流）** | 可复用的工作流片段 | "AI 文本摘要"子工作流可在多处调用 |

### 3.3 安装与配置

```bash
# 方式1：Docker 一键部署（推荐）
docker run -d \
  --name n8n \
  -p 5678:5678 \
  -v ~/.n8n:/home/node/.n8n \
  n8nio/n8n

# 方式2：npm 直接安装
npm install n8n -g
n8n start

# 访问：http://localhost:5678
# 首次访问需要创建管理员账号
```

### 3.4 项目示例：用 n8n 搭建"AI 自动生成周报"工作流

**场景**：每周末，AI 自动从你的 Jira 任务、GitHub 活动、日历事件中提取信息，生成一份团队周报。

**工作流设计**：

```
┌──────────────┐    ┌─────────────────┐    ┌────────────────┐
│              │    │                 │    │                │
│ Schedule     │ → │ Jira: 获取本周  │ → │ GitHub: 获取   │
│ (每周五 17:00)│    │ 已完成任务      │    │ 本周合并的 PR  │
│              │    │                 │    │                │
└──────────────┘    └─────────────────┘    └────────────────┘
                           │                       │
                           └───────────┬───────────┘
                                       │
                              ┌────────▼────────┐
                              │                  │
                              │ AI 节点:         │
                              │ "将以下信息生成   │
                              │  一份简洁的周报，  │
                              │  按项目分组"      │
                              │                  │
                              └────────┬─────────┘
                                       │
                              ┌────────▼────────┐
                              │                  │
                              │ Slack: 发送到    │
                              │ #team-weekly     │
                              │                  │
                              └────────┬─────────┘
                                       │
                              ┌────────▼────────┐
                              │                  │
                              │ Email: 抄送      │
                              │ 经理和团队        │
                              │                  │
                              └─────────────────┘
```

**n8n 节点的实际配置**：

1. **Schedule Trigger**：每周五 17:00

```json
{
  "node": "Schedule Trigger",
  "parameters": {
    "rule": {
      "interval": 0,
      "cronExpression": "0 17 * * 5"
    }
  }
}
```

2. **Jira Node**：获取本周已完成任务

```json
{
  "node": "Jira",
  "parameters": {
    "operation": "search",
    "jql": "assignee = currentUser() AND status = Done AND updated >= startOfWeek()",
    "fields": ["summary", "status", "updated"]
  }
}
```

3. **GitHub Node**：获取本周合并的 PR

```json
{
  "node": "GitHub",
  "parameters": {
    "operation": "listPullRequests",
    "repository": "my-org/my-project",
    "filters": {
      "state": "merged",
      "since": "{{ $now.startOf('week').toISO() }}"
    }
  }
}
```

4. **AI 节点**：生成周报内容

```json
{
  "node": "OpenAI (or Claude)",
  "parameters": {
    "prompt": "你是一个项目经理。根据以下数据生成一份周报：\n\nJira 任务：{{ $json.jiraTasks }}\nGitHub PR：{{ $json.githubPRs }}\n\n格式：\n## 本周完成\n- [项目名] 具体内容\n\n## 下周计划\n- \n\n## 阻塞项\n- ",
    "model": "gpt-4o"
  }
}
```

5. **Slack Node + Email Node**：发送周报

```json
{
  "node": "Slack",
  "parameters": {
    "channel": "#team-weekly",
    "text": "{{ $json.aiReport }}"
  }
}
```

**结果**：每周五下班前，你的周报自动出现在团队 Slack 频道和经理邮箱里。**你不需要任何操作。**

### 3.5 n8n 的 AI 特有节点

n8n 提供了一系列 AI 原生节点，让你的工作流不只是"触发→发送"，而是"触发→AI 处理→决策→执行"：

| AI 节点 | 功能 | 应用场景 |
|:--------|:-----|:---------|
| **LLM** | 调用大语言模型（支持 Claude, GPT, 本地模型） | 文本生成、分类、提取 |
| **AI Agent** | 完整的 AI Agent 节点，带工具调用能力 | 自主完成复杂任务 |
| **Embeddings** | 文本向量化 | 知识库搜索、相似度匹配 |
| **Vector Store** | 向量数据库操作 | RAG 检索 |
| **Text Classifier** | 文本分类 | 邮件分类、工单分流 |
| **Sentiment Analysis** | 情感分析 | 客户反馈分析 |
| **Extract Structured Data** | 从非结构化数据提取结构化信息 | 从邮件提取订单信息 |

## 🍅 番茄104（T4）：Dify 与 Coze——低代码 AI 应用平台

### 4.1 什么是低代码 AI 平台？

低代码 AI 平台让你**不用写代码就能搭建 AI 应用**。你不是"开发"AI 应用，而是"搭建"AI 应用——拖拽、配置、连接。

```
传统 AI 应用开发：                   低代码 AI 平台：
写代码 → 搭环境 → 部署 → 运维         拖拽 → 配置 → 发布
    ↓                                       ↓
3-6 周                                  30 分钟
专业开发团队                             产品经理也能做
```

**两大代表**：

| 维度 | Dify | Coze |
|:-----|:-----|:------|
| **出品方** | 开源社区（可自托管） | 字节跳动 |
| **开源** | ✅ 完全开源 | ❌ 闭源 |
| **部署** | 本地 / 云端 | 云端 SaaS |
| **核心特长** | RAG 应用、工作流 | Bot 分发、插件生态 |
| **适合** | 企业级 AI 应用 | 社交/内容 AI Bot |

### 4.2 Dify——开源的"AI 应用工厂"

Dify 是一个开源的低代码 AI 应用开发平台。它最擅长的两件事：
1. **RAG（检索增强生成）**——让 AI 基于你的知识库回答问题
2. **AI 工作流**——可视化的 AI 处理管道

### 4.3 项目示例：在 Dify 上搭建一个 RAG 聊天机器人

**场景**：你的公司有一堆产品文档（PDF、Word、Markdown），你希望员工可以在一个聊天界面里问"我们的 API 限流策略是什么？"并得到答案。

**步骤 1：创建知识库**

```
1. 登录 Dify → 知识库 → 创建知识库
2. 上传所有产品文档（支持批量上传）
3. 选择分段策略：
   - 分段方式：Markdown 标题分段
   - 分段长度：500 tokens
   - 重叠长度：50 tokens（保持上下文连贯）
4. 选择 Embedding 模型：text-embedding-3-small
5. 点击"处理并导入"
```

**步骤 2：创建 AI 助手**

```
1. 工作室 → 创建应用 → 聊天助手
2. 配置助手：
   - 名称：产品知识助手
   - 模型：Claude 3.5 Sonnet
   - 提示词：
     "你是公司产品知识助手。基于提供的知识库回答员工问题。
     如果知识库中没有相关信息，请诚实地说'这个问题我无法从现有文档中找到答案'。
     引用来源文档名称。"
   - 添加上一步创建的知识库
3. 设置：
   - 记忆：开启（记住对话历史）
   - 引用：显示（在回复中标注引用来源）
   - 审核：敏感词过滤
```

**步骤 3：发布**

```
1. 预览测试：
   Q: "API 限流策略是什么？"
   A: "根据《API 开发文档》第3章，我们的 API 限流策略如下：
       - 免费用户：每分钟 60 次
       - 付费用户：每分钟 1000 次
       - 超出限流返回 429 状态码
       [引用：API开发文档.pdf 第12页]"

2. 发布为：
   - Web 页面（生成链接）
   - API（供其他系统调用）
   - 嵌入代码（放到公司内部网站）
```

### 4.4 Coze——字节跳动的"Bot 分发平台"

Coze 的特色不仅在于"搭建"AI Bot，更在于"分发"——让你的 Bot 出现在微信、飞书、抖音、Web 等各种渠道。

**Coze 的核心能力**：

| 能力 | 说明 | 例子 |
|:-----|:------|:------|
| **Bot 搭建** | 可视化配置 AI 助手 | "设定角色、知识库、技能、对话风格" |
| **插件生态** | 200+ 预置工具 | 新闻搜索、图片生成、天气查询 |
| **知识库** | 上传文档向量化 | 公司手册、产品文档 |
| **工作流** | 多步骤 AI 处理 | 先分类→再回答→最后记录 |
| **多渠道分发** | Bot 发布到多个平台 | 微信、飞书、抖音、API |
| **Bot 商店** | 共享和发布 Bot | 开箱即用的 AI 工具 |

**Coze 典型应用**：

- **客服 Bot**：连接产品知识库，在微信公众号自动回复客户问题
- **内容助手**：根据你的文档风格，自动生成推文/文章初稿
- **数据助手**：连接数据库，用自然语言查询数据
- **学习助手**：上传学习资料，搭建私人的学习问答 Bot

> ✋ **费曼自测**：Dify 和 Coze 的目标用户有什么不同？如果你的公司想搭建内部知识库问答系统，你会选哪个？为什么？

### 5.2 工具的"三维决策矩阵"

面对这么多工具，怎么选？从三个维度分析：

| 你要解决什么问题？ | 推荐工具 | 一句话理由 |
|:------------------|:---------|:-----------|
| **让 AI 连接外部工具** | **MCP 协议** | 它是标准，不是工具——所有工具都通过它连接 |
| **搭建多步骤自动化** | **n8n** | 可视化+开源+400+集成+AI原生 |
| **搭建企业 AI 应用** | **Dify** | 开源+RAG+工作流，适合私有化部署 |
| **快速铺到社交渠道** | **Coze** | 微信/飞书/抖音一键发布 |
| **编排多个 AI 协作** | **OpenClaw** | 多模型路由+持久Agent+人类介入 |
| **自动化个人办公** | **n8n + OpenClaw** | n8n 做流程，OpenClaw 做 AI 决策 |
