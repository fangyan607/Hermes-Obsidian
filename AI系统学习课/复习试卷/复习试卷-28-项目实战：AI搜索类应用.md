# 复习试卷-28-项目实战：AI搜索类应用

---

## Section A：费曼学习法复习文档

### 1. 核心概念：企业AI搜索问答的三大需求

- **响应速度**：首个 token 回复需在 **3-5秒内**
- **准确率**：客服辅助场景要求 **99.9%以上**，可以不回答，但不能出错
- **数据规模**：文档通常 100+，互联网企业可达上亿级别

### 2. RAG的三级架构（Qwen-Agent）

| 级别 | 名称 | 核心思路 |
|------|------|----------|
| Lv1 | 检索 | 关键词提取 + BM25 检索。将用户查询拆分为"信息"和"指令"，再从信息部分推导多语言关键词，用 BM25 匹配 |
| Lv2 | 分块阅读 | 暴力策略：对每个 512 字块让 LLM 评估相关性，并行处理，过滤出相关句子后再用 BM25 检索 |
| Lv3 | 逐步推理 | 将 Lv2 封装为工具，由 Lv3 智能体调用进行多跳推理（如：第五交响曲 → 19世纪 → 交通工具） |

### 3. Qwen-Agent 检索实现的关键文件

- 文档拆分：`qwen_agent/tools/simple_doc_parser.py`
- 检索逻辑：`qwen_agent/tools/retrieval.py`
- 内存管理：`qwen_agent/memory/memory.py`（加载 retrieval 和 doc_parser 两个工具）
- 数据存储位置：**内存中**，数据是临时的，仅为单次查询服务，不会被持久化

### 4. 文档解析与分块流程

1. retrieval 工具调用 doc_parser 处理文件
2. doc_parser 根据文件扩展名（.pdf, .docx, .txt 等）自动选择解析器
3. 按 `parser_page_size`（默认 500 字符）将文本切分成 **chunks**
4. 每个 chunk 格式：`{'page_content': '...', 'metadata': {...}}`

### 5. BM25 得分计算的三个因素

- **词频（TF）**：查询词在文档中出现次数越多，得分越高
- **逆文档频率（IDF）**：词越稀有，权重越高（如"保险"权重低，"高空坠物责任"权重高）
- **字段长度归一化**：字段越短，得分越高（短文档中匹配到关键词更能体现重要性）

### 6. Elasticsearch 核心要点

- **定义**：开源的高扩展分布式全文检索引擎
- **基于 Java 开发**，从 ES 7.0 开始安装包自带 OpenJDK
- 单文档上限：**2,147,483,519**
- 单机检索性能：**0.005~5.6秒**，可满足 95%+ 业务场景
- **BM25 算法**：综合考虑 TF、IDF、字段长度计算相关度得分

### 7. Elasticsearch 搜索流程

1. 分析查询，提取关键词（如"工伤保险"、"雇主险"、"区别"）
2. 遍历索引中的每个文档
3. 对每个文档，综合 TF、IDF、字段长度计算分数
4. 合并所有关键词分数，得出最终 `_score`

### 8. Qwen3-Embedding

- **多语言支持**：覆盖 119 种语言，MTEB 多语言榜单第一（70.58分）
- **长文本处理**：支持 32K 上下文
- **灵活维度**：可输出 32~4096 维向量
- **版本**：0.6B / 4B / 8B
- 调用方式：通过 DashScope API（`text-embedding-v4` 模型）

### 9. Gradio 界面关键概念

- **Blocks API**：低级布局 API，允许像"搭积木"一样自由组合 UI 组件
- 布局组件：`Row`（水平）、`Column`（垂直）、`Tab`（标签页）、`Accordion`（折叠面板）、`Group`（逻辑组）

### 10. 关键 Agent 类型与用途

| Agent | 文件位置 | 用途 |
|-------|----------|------|
| `DialogueRetrievalAgent` | `qwen_agent.agents` | 超长对话/文本检索问答 |
| `ParallelDocQA` | `qwen_agent.agents.doc_qa` | 多文档/大文档并行问答 |
| `ReActChat` | `qwen_agent.agents` | 工具调用+推理（代码解释、数据分析） |
| `BasicDocQA` | `qwen_agent.agents.doc_qa` | 文档问答 |
| `Assistant` | `qwen_agent.agents.assistant` | 通用助手，集成 RAG + 工具调用 |
| `Router` | `qwen_agent.agents` | 多智能体自动路由 |

### 11. `rag_cfg` 参数详解

- `max_ref_token`：单次最大参考 token 数，默认 20000
- `parser_page_size`：文档分片每片最大 token 数，默认 500
- `rag_keygen_strategy`：关键词生成策略（`GenKeyword` / `SplitQueryThenGenKeyword` / `GenKeywordWithKnowledge`）
- `rag_searchers`：检索子策略列表，如 `['keyword_search', 'front_page_search']`

### 12. ParallelDocQA 核心流程

1. **文件解析与分块**：提取文件 → 按 chunk size（如 1000 tokens）分块
2. **并行分片问答**：通过 `parallel_exec` 对每个 chunk 独立调用 LLM 问答
3. **结果收集与过滤**：过滤无效/空响应
4. **检索与摘要**：生成关键词 → 对原始文档再次检索 → 摘要 Agent 生成最终答案

#### 问答题参考答案

**第1题：**

**Lv1（检索）**：将用户查询拆分为"信息"和"指令"两部分，从信息部分推导多语言关键词，用 BM25 算法找出最相关的文本块。

**Lv2（分块阅读）**：对每个 512 字块让 LLM 评估相关性（并行处理），过滤出"相关"的句子，再用 BM25 检索最相关的块（总结果控制在 8K token 内）。

**Lv3（逐步推理）**：将 Lv2 智能体封装为一个工具，由 Lv3 工具调用智能体调用。当需要多跳推理时（如"与第五交响曲同世纪的交通工具是什么"），Lv3 先提出子问题（"第五交响曲在哪个世纪创作"）→ 调用 Lv2 获取答案（"19世纪"）→ 再提出下一个子问题（"19世纪发明了什么交通工具"）→ 最终整合所有反馈回答原始问题。

**第2题：**

**传统DocQA/RAG**：先检索少量相关片段，再让 LLM 生成答案。如果检索阶段召回不充分，后续回答受限于有限的上下文，结果不全或不精准。

**ParallelDocQA**：将文档分成大量小片段（chunk），对每个片段独立发起一次 LLM 问答（并行处理），极大提升召回覆盖面。之后过滤无效响应，用有效片段生成更精准的检索关键词，对原始文档再次检索（RAG召回），最后用摘要 Agent 综合归纳。

**核心优势**：
1. 并行分片问答提升召回覆盖率，不遗漏重要信息
2. 多轮过滤与再检索提升精准度
3. 适合大文档/多文档/复杂结构场景
