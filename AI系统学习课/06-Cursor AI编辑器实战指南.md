## 番茄钟1：理解Cursor的核心思维模型（25分钟）

### 1.1 一句话说清Cursor是什么

**用大白话讲：**
Cursor 是一个把 AI 直接塞进编辑器里的工具。它不是一个"帮你写代码的插件"，而是**一个会自己读代码、想方案、动手改代码的编程助手（Agent）**。

你可以把它想象成一个坐在你旁边、随时可以问问题的初级程序员同事——你告诉它目标，它会自己去翻代码、找文件、写代码、跑命令。

**核心思维转变：**
```
传统写代码：我构思 → 我写 → 我改 → 我查
Cursor 写代码：我描述目标 → Cursor 理解 → Cursor 执行 → 我审核
```

> ✋ **费曼自测**：停一下，用自己的话解释"Cursor 和 VS Code Copilot 有什么区别？"如果说不清楚，回头重读上一段。

---

### 1.2 Cursor 的三种工作模式

Cursor 根据你要做的事情，提供三种不同的交互方式：

| 模式              | 触发方式                        | 适用场景        | 一句话理解           |
| --------------- | --------------------------- | ----------- | --------------- |
| **Chat**        | `Cmd+I`（Mac）/ `Ctrl+I`（Win） | 问答、解释代码、小修改 | 像 ChatGPT 一样问问题 |
| **Composer**    | `Cmd+Shift+I`               | 多文件、复杂功能开发  | 一次性帮你搞定一个完整功能   |
| **Inline Edit** | 选中代码后 `Cmd+K`               | 改当前这一段代码    | 选中一段，告诉它怎么改     |

**选择口诀：**
- 想问东西 → Chat
- 改小东西 → Inline Edit
- 做大东西 → Composer / Agent

---

### 1.3 Agent 模式——Cursor 最强大的功能

Agent 模式是 Cursor 的核心能力。它不是简单地补全代码，而是**自主完成编程任务**。

**Agent 的工作原理（三个齿轮）：**

```
指令（Rules + 提示词）
    ↓
工具（文件读写 + 搜索 + 终端命令 + 浏览器）
    ↓
模型（Claude / GPT / Gemini 等）
```

**Agent 拥有的工具（类比人类工具包）：**

| 工具 | 类比 | 解释 |
|------|------|------|
| 语义搜索 | 代码版 Google | 按意思找代码，不只是搜关键词 |
| 文件读写 | 你的手和眼 | 自动读文件、改文件 |
| 终端命令 | 你的命令行 | 自动运行安装、测试、构建命令 |
| 浏览器 | 你的屏幕 | 打开网页截图，验证 UI 是否正确 |
| 提问题 | 你的嘴 | 遇到不清楚的会主动问你 |

**关键概念：检查点（Checkpoints）**
Agent 在做出重大修改前会自动拍一张代码"快照"。如果改坏了，可以一键回滚。和 Git 是独立的，专门用来撤销 Agent 的改动。

> ✋ **费曼自测**：想象你要向一个完全不懂编程的朋友解释"Agent 模式到底在做什么"。用不超过三句话说出来。

### 1.4 Rules——让AI按你的规矩办事

**为什么需要 Rules？**
LLM（大语言模型）没有记忆。每次对话它都"失忆"，不知道自己之前做过什么。Rules 就是在每次对话开始时告诉它："记住，这是规矩。"

**四种 Rules：**

| 类型 | 存哪里 | 谁可用 |
|------|--------|--------|
| **Project Rules (.mdc)** | `.cursor/rules/` 目录下 | 团队共享，跟着项目走 |
| **User Rules** | Cursor 设置里 | 只有你自己 |
| **Team Rules** | 团队 Dashboard | 企业团队 |
| **AGENTS.md** | 项目根目录 | 简单好用的替代方案 |

**如何创建一条 Rule？**
在 Chat 里输入 `/create-rule`，然后描述你想要的规则。比如：

```
/create-rule
我想要一条规则：所有 API 请求必须包含错误处理，
每个 try-catch 都要有具体的错误类型判断。
```

**Rule 的文件结构（.mdc 格式）：**
```markdown
---
description: API 请求必须包含完整的错误处理
globs: src/**/*.ts
alwaysApply: true
---

# API 错误处理规范

所有 `fetch` 或 `axios` 调用必须：
1. 使用 try-catch 包裹
2. catch 中区分网络错误 vs 业务错误
3. 错误信息必须有中文提示语
```

**规则类型选择逻辑：**
- 全局通用的 → User Rules
- 项目通用的 → AGENTS.md（最简单）
- 需要精细控制的 → `.cursor/rules/*.mdc`（支持 glob 匹配特定文件）
- 团队强制执行 → Team Rules

> ✋ **费曼自测**：用自己的话描述"Rules 解决了什么问题？为什么不能直接把要求写在提示词里？"

## 番茄钟2：进阶功能与实战流程（25分钟）

### 2.1 MCP——让AI连接外部世界

**用大白话讲：**
MCP（Model Context Protocol）是一个"万能接口"。它让 Cursor 的 AI 能直接连接你用的其他工具——数据库、Jira、GitHub、Slack 等。

**没有 MCP 时：**
```
你在 Cursor 里 → 切到浏览器查 Jira → 切回 Cursor 写代码
```

**有 MCP 时：**
```
你在 Cursor 里说"帮我查 Jira 上这个 ticket 的详情"
                ↓
        AI 通过 MCP 直接查 Jira API
                ↓
        结果回到 Cursor 里，AI 直接据此写代码
```

**三种连接方式：**

| 方式 | 一句话理解 | 适合 |
|------|------------|------|
| `stdio` | 本地启动一个命令行工具 | 个人使用 |
| `SSE` | 启动一个 HTTP 服务 | 多人共用 |
| `Streamable HTTP` | 标准 HTTP 接口 | 远程服务 |

**如何配置 MCP 服务器？**
在项目根目录创建 `.cursor/mcp.json`：

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${env:GITHUB_TOKEN}"
      }
    }
  }
}
```

配置好后，Agent 在对话中会自动判断是否需要调用这些工具。

### 2.2 Skills——可复用的AI能力包

**用大白话讲：**
Skills 就是把"你经常让 AI 做的事"打包成一个可共享的文件。比如"帮我写测试""帮我做代码审查"，封装好之后一键复用。

**Skills vs Rules 的区别：**

```
Rules：告诉 AI "你的规矩是什么"（被动约束）
Skills：告诉 AI "你会什么活"（主动能力）
```

**一个 Skill 长什么样？**
```
.skills/
└── test-generator/      ← Skill 文件夹
    ├── SKILL.md         ← Skill 定义文件（必须）
    ├── scripts/         ← 辅助脚本（可选）
    └── references/      ← 参考文档（可选）
```

**Skill 的定义文件（SKILL.md）：**
```markdown
---
name: test-generator
description: 生成符合项目规范的单元测试，支持 Jest 和 Vitest
paths: src/**/*.ts
---

# 测试生成 Skill

当要求生成测试时，遵循以下步骤：
1. 读取源文件，理解导出内容
2. 按项目测试规范生成测试文件
3. 确保覆盖正常路径和边界情况
```

**已安装的 Skills 自动在 Agent 输入 `/` 时显示。**
你只需要输入 `/` 就能看到可用命令列表。

### 2.3 Plan Mode——先想清楚再做

**为什么需要 Plan Mode？**
AI 经常"想太多"——你让它改个按钮颜色，它把整个页面重写了。Plan Mode 就是让 AI **先出方案，你审核通过后它再动手**。

**Plan Mode 的工作流程：**

```
你提需求 → AI 出方案（不改代码） → 你审核 → 确认执行 → AI 动手改
                                    ↑ 关键步骤
```

**实操技巧：**
- 复杂功能开发时，**先切到 Plan Mode** 让 Agent 出设计方案
- 审核方案中的文件列表、修改范围
- 确认后再让 Agent 执行
- 这能避免 80% 的"AI 改多了"的困扰

### 2.4 日常工作流实战

**场景 A：接手一个陌生项目（理解代码）**
```
1. Agent 模式 → "帮我理解这个项目的架构"
2. AI 会自己读代码、画架构图、总结目录结构
3. 你只需要看它给你的总结即可
```

**场景 B：修一个 Bug（调试）**
```
1. Agent 模式 → "这个按钮点了没反应，帮我看为什么"
2. AI 会搜索相关代码、分析逻辑、定位问题
3. 改完后它会问"要不要验证？"→ 确认后自动运行测试
```

**场景 C：做一个完整功能（从方案到上线）**
```
Step 1： Plan Mode → "帮我设计用户登录功能方案"
Step 2： 你审核方案，调整修改范围
Step 3： Agent 执行 → 创建文件、写代码、跑测试
Step 4： Agent Review → AI 自己审查自己的改动
```

### 2.5 最佳实践清单

**✅ 应该做的：**
- 启动新项目时，先写一个 `AGENTS.md` 放项目根目录
- 复杂功能先切 Plan Mode，不要直接让 AI 动手
- 关键 Rules 放到 `.cursor/rules/` 下，用 `/create-rule` 生成
- 使用 Checkpoint 功能，每次大改前拍个快照
- 在 Agent 工作时可以继续打字排队消息，不用等

**❌ 避免做的：**
- 不要把"写测试"这种通用命令写在项目 Rules 里——应该用 Skills
- 不要在 Rules 里重复 linter 规则（AI 会自动遵守 linter）
- 不要写超过 500 行的 Rules——拆成多个小 Rules
- 不要过度相信 AI——每次修改后都要审核 diff
