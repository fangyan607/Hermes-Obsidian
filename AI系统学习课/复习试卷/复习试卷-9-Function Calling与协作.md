| 维度 | Function Calling | MCP |

| 适用场景 | 简单单次调用 | 多轮复杂上下文 |

## 💡 阶段二：机制提取

### 核心问题链

**1. Function Calling是什么？解决什么问题？**
Function Calling是大模型调用外部函数的能力。它解决了LLM的三大局限：①无法获取实时数据（如天气、股价）；②无法执行精确计算（复杂数学运算）；③无法操作外部系统（发邮件、订票等）。通过Function Calling，模型可以将自然语言转化为结构化的函数参数，并调用对应函数执行。

**2. Function Calling的3大作用分别是什么？**
- **扩展模型能力**：获取实时数据（天气API）、执行复杂计算（数学函数）、操作外部系统（数据库/邮件）
- **结构化输出**：将模糊的自然语言（"查北京天气"）转化为精确的函数参数`{city: "北京", type: "base"}`
- **动态决策流程**：链式调用多个函数，模型自主规划执行流程（先查天气→再推荐活动→最后生成方案）

**3. Function Calling和MCP的核心区别是什么？**
- **性质不同**：FC是模型厂商私有接口（如OpenAI的Function Calling），MCP是开放协议（Anthropic推出）
- **通用性不同**：FC绑定特定模型，换模型需重写；MCP跨模型通用，一次开发到处用
- **适用场景不同**：FC适合简单单次调用（查个天气），MCP适合多轮复杂上下文（多工具协作）
- **发展方向**：FC是当前方案，MCP是未来趋势——从"各自为政"走向"统一标准"

**4. Qwen3模型的架构设计有什么特点？**
- **双架构设计**：MoE（混合专家）+ Dense（稠密），满足不同需求
- **MoE旗舰**：Qwen3-235B-A22B，总参数235B但只激活22B，推理效率极高
- **MoE高效**：Qwen3-30B-A3B，总参数30B只激活3B，轻量高效
- **6档Dense模型**：从0.6B到32B，覆盖从手机端到服务器端的部署需求
- **Apache 2.0开源**：商业友好，促进生态发展

**5. 天气调用CASE的4步流程是怎样的？**
1. **设置API Key**：`amap_api_key = "xxx"`，配置高德地图API密钥
2. **定义Function Tool**：用JSON Schema描述`get_weather`函数，包含函数名、描述、参数（city、type）及类型约束
3. **实现工具函数**：`def get_weather(city, type="base")`，调用高德天气API返回真实天气数据
4. **主流程run_weather_query**：将用户问题和tools定义发送给模型 → 模型返回tool_calls → 执行对应函数 → 将函数结果返回模型 → 模型生成最终回答

**6. tool_choice="auto"的作用是什么？**
`tool_choice="auto"`表示模型自动决定是否调用工具。当用户问题不需要工具时（如"你好"），模型直接回答；当需要工具时（如"北京天气"），模型自动生成tool_calls。这避免了每次都强制调用工具的浪费，也防止了不需要工具时错误调用。

**7. 门票助手案例的核心设计思路是什么？**
门票助手展示了Function Calling在实际业务中的应用：定义景点查询、门票预订、订单查询等多个工具函数，模型根据用户意图自动选择调用哪个（或哪几个）函数，实现多步骤业务流程的自动化。核心是"自然语言意图 → 结构化函数调用 → 业务系统执行"的转换链路。

## 🗣️ 阶段三：简化讲解

### 给新人讲解版

> 今天我要给你讲一个让AI"从只会聊天变成能干活"的技术——Function Calling。
>
> **是什么**：AI模型本来只会"说话"，不能查天气、不能算数学、不能订票。Function Calling就是给AI装上"外挂"——让它能在需要的时候"打电话"给外部工具，获取真实数据、执行真实操作。
>
> **3大作用**：第一，扩展能力——AI自己不知道今天天气，但可以调天气API查；第二，结构化输出——你说"查北京天气"，AI能把"北京"精准提取出来作为参数；第三，动态决策——AI能自己规划"先查天气→再推荐景点→最后生成攻略"的流程。
>
> **和MCP什么关系**：Function Calling像是"每个手机厂商自己的充电线"，MCP像是"USB-C统一标准"。FC简单好用但绑定特定模型，MCP更通用但搭建稍复杂。简单任务用FC，复杂协作用MCP。
>
> **Qwen3是什么**：阿里2025年4月发布的最新开源大模型。有两种设计——MoE模型像"请了235个专家但每次只叫22个"，省钱又高效；Dense模型像"全员上班"，从0.6B到32B覆盖各种设备。全部Apache 2.0开源，商用也没问题。
>
> **天气调用怎么实现**：四步走——第一步配密钥，第二步告诉AI"我有个get_weather函数，参数是city"，第三步写好真正的天气查询代码，第四步把用户问题和工具定义一起发给AI，AI决定调就调，不调就直答。
>
> **tool_choice="auto"**：就是让AI自己决定要不要用工具。问"你好"就直答，问"天气"就调API，聪明又省事。

### 可视化类比表

| 抽象概念 | 具体类比 | 为什么这样类比 |
|----------|----------|---------------|
| Function Calling | 给AI装外挂 | AI本身能力有限，外挂扩展能力 |
| 结构化输出 | 翻译官把模糊需求变成精确订单 | "查北京天气"→`{city:"北京"}` |
| 动态决策流程 | AI自己规划工作步骤 | 先查天气→再推荐→最后总结 |

| Qwen3 Dense模型 | 全员上班 | 全参数激活，简单直接 |
| tool_choice="auto" | 智能开关 | AI自己判断需不需要用工具 |
| 天气API | 打电话问气象局 | AI不知道天气但可以查 |

## ✅ 阶段四：验证练习

### 基础题

**题目1**：Function Calling的3大作用是什么？请分别说明。

> 解答：
> 1. **扩展模型能力**：获取实时数据（天气/股价）、执行复杂计算、操作外部系统，突破LLM自身的局限
> 2. **结构化输出**：将自然语言转化为结构化的函数参数，如"查北京天气"→`{city: "北京"}`
> 3. **动态决策流程**：链式调用多个函数，模型自主规划多步骤执行流程

**题目2**：请说明Function Calling与MCP的3个核心区别。

> 解答：
> 1. **性质不同**：FC是模型厂商私有接口，MCP是开放协议标准
> 2. **通用性不同**：FC绑定特定模型，换模型需重写；MCP跨模型通用
> 3. **适用场景不同**：FC适合简单单次调用，MCP适合多轮复杂上下文

**题目3**：Qwen3的MoE模型和Dense模型有什么区别？

> 解答：
> - **MoE模型**：参数总量大但激活量小，如Qwen3-235B-A22B有235B参数但只激活22B，推理效率高
> - **Dense模型**：全参数激活，从0.6B到32B覆盖不同资源场景
> - MoE适合追求高性能低成本的场景，Dense适合资源确定、需要稳定性能的场景

### 变式题

**题目4**：在天气调用CASE中，为什么要用JSON Schema来描述函数，而不是直接告诉模型"你可以在需要时调用get_weather函数"？

> 解答：JSON Schema提供了精确的函数签名，包括函数名、参数名、参数类型、必填/选填、描述等信息。如果只口头描述，模型可能：①无法生成正确的参数格式；②遗漏必填参数；③参数类型错误。JSON Schema让模型能精确地生成结构化的tool_calls，确保函数可执行。

**题目5**：tool_choice参数除了"auto"之外，还可能有哪些取值？它们分别适用于什么场景？

> 解答：
> - **"auto"**：模型自动决定是否调用工具，适用于大多数通用场景
> - **"none"**：禁止模型调用任何工具，适用于纯对话场景（如创意写作）
> - **"required"**：强制模型必须调用至少一个工具，适用于必须获取外部数据的场景
> - **指定函数名**：强制模型调用特定函数，适用于确定性流程场景

**题目6**：请说明门票助手案例中，Function Calling如何实现"自然语言意图→业务系统执行"的转换，并分析这种设计模式的优势。

> 解答：
> - **转换过程**：用户说"帮我订两张故宫门票"→ 模型识别意图为"订票"→ 生成tool_calls调用book_ticket函数，参数`{attraction: "故宫", count: 2}`→ 执行函数返回订单结果 → 模型生成自然语言回复
> - **优势**：①用户体验好（自然语言交互）；②解耦设计（模型负责理解，函数负责执行）；③可扩展（新增业务只需定义新函数）；④可追踪（每次函数调用都有记录）；⑤准确性高（结构化参数避免歧义）

### 三、问答题答案要点

**1. Function Calling 3大作用详解：**

**作用1：扩展模型能力**
- 没有FC时：模型只能基于训练数据回答，无法获取实时信息，回答"今天天气"只能编造或说"我不知道"
- 有FC后：模型调用天气API获取真实数据，回答准确可靠
- 场景示例：股票行情查询——用户问"苹果股价多少"，模型调用股价API返回实时数据

**作用2：结构化输出**
- 没有FC时：模型输出自然语言文本，格式不可控，无法直接传递给下游系统
- 有FC后：模型输出结构化JSON参数，可直接作为函数入参执行
- 场景示例：订餐系统——用户说"我要一份麻辣香锅微辣"，模型输出`{dish: "麻辣香锅", spice: "微辣"}`，厨房系统直接接收处理

**作用3：动态决策流程**
- 没有FC时：模型只能一次性生成文本，无法根据中间结果调整策略
- 有FC后：模型可以链式调用多个函数，根据每步结果决定下一步操作
- 场景示例：旅行规划——用户说"帮我规划周末出行"，模型先查天气→根据天气推荐景点→查门票价格→生成完整方案

**2. Function Calling vs MCP对比：**

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| 性质 | 模型厂商私有接口 | 开放协议（标准） |
| 通用性 | 绑定特定模型，换模型需重写 | 跨模型、跨平台通用 |
| 适用场景 | 简单单次调用 | 多轮复杂上下文 |
| 开发复杂度 | 低（直接定义函数即可） | 中（需搭建MCP Server） |
| 复用性 | 低（绑定模型，不跨平台） | 高（一次开发多场景复用） |

**选择Function Calling的情况：**
- 快速原型验证，需要最小开发成本
- 只需调用1-2个简单函数（如查天气、算数学）
- 明确只使用一个模型，不考虑跨模型兼容
- 项目周期短，不需要长期维护

**选择MCP的情况：**
- 生产环境部署，需要稳定可维护的架构
- 需要集成多个工具和数据源
- 需要跨模型兼容（今天用GPT，明天可能换Claude）
- 多Agent协作场景，需要标准化的通信协议
- 长期项目，需要考虑生态和可扩展性
