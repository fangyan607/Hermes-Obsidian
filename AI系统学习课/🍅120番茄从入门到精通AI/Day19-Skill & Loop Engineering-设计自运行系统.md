# Day19：Skill & Loop Engineering——设计自运行系统（番茄91-95）

> **Part 4 · 大语言模型与智能体 · 终章**
> **悬疑探案进度**：从"手动提问"到"设计自运行系统"——你的角色已经彻底改变。
> **番茄范围**：🍅91-95

---

## 悬疑开场：2026年，最好的AI工程师不再写Prompt

> *"I don't prompt Claude anymore. I have a bunch of loops running that will prompt Claude and decide what to do next on their own. My job is to write loops."*
> — Boris Cherny, Anthropic Claude Code 负责人

这句话标志着一个时代的转折点。

**2023年**：最强的AI技能是"写Prompt"——你要知道怎么问，AI才会好好答。

**2024年**：最强的AI技能是"设计Context"——你要知道给AI什么资料，它才能好好干。

**2025年**：最强的AI技能是"搭Harness"——你要知道给AI什么工具环境。

**2026年**：最强的AI技能是**"设计Loop"**——你要设计一个系统，让AI自己跑起来。

你不是"操作员"了。你是**架构师**。

今天——Part 4的终章——你将完成从"Prompt写手"到"Loop架构师"的转变。

## 🍅 番茄91 · T1：从Prompt到Loop的范式转移

### 🔍 犯罪现场：你的日常已经是一个Loop

每天早上你打开电脑：

1. 检查消息 → 发现Issue #42有人报Bug
2. 分析问题 → 定位到代码中的问题
3. 写修复 → 编译 → 发现没通过
4. 再改 → 再编译 → 通过了
5. 提PR → 等Review → 合并

这就是一个**Loop**——你用你的大脑当"控制器"。

**Loop Engineering要做的，就是把你的大脑从"控制器"升级为"系统设计师"。**

### 🧠 四代范式进化史（回顾）

```
Prompt Engineering (2023)
  问题："怎么写指令AI才听？"
  角色：操作员（写Prompt → 看回复 → 再写）
  
Context Engineering (2024)
  问题："给AI配什么资料？"
  角色：资料员（选文档 → 配上下文 → 优化检索）

Harness Engineering (2025)
  问题："给AI搭什么工具环境？"
  角色：环境搭建者（配工具 → 配权限 → 配安全）

Loop Engineering (2026)
  问题："设计什么系统让AI自己跑？"
  角色：系统架构师（设计循环 → 定停止条件 → 监控运行）
```

**每次进化都不是取代上一代——而是建立在上一代之上。**

### 🧠 关键转变：从"Human-in-the-Loop"到"Human-as-Architect"

```
                   2023                    2026
            Human-in-the-Loop        Human-as-Architect
                  │                         │
                  ▼                         ▼
        你：写代码实现X           你：定义X的质量标准
        AI：生成代码               AI：自己写 → 自己测
        你：检查代码                    → 自己改 → 直到达标
        你：说"这里改一下"
        AI：改
        你：再说"这里也要改"
        ...
```

| 维度 | Human-in-the-Loop | Human-as-Architect |
|:-----|:-----------------|:-------------------|
| 你在做什么 | 每一步手动推进 | 设计系统让它自己推进 |
| 你和AI的关系 | 你是操作员，AI是工具 | 你是设计师，AI是执行者 |
| 你的关注点 | "这句话怎么说AI才懂？" | "这个循环怎么设计才可靠？" |
| 出错了谁负责 | 你（每一步都是你确认的） | 你（但你需要设计错误恢复） |
| 可扩展性 | 低（你的时间 = 系统的产出） | 高（系统24小时自动运行） |

> ✋ **费曼自测**：向一个2019年的程序员解释"Human-as-Architect"这一概念。他从来没听说过AI Agent。用"自动驾驶"做类比。

### 🧠 核心洞察：Loop不是一种技术，是一种思维方式

Loop Engineering的底层哲学：

```
1. 一切重复性工作都可以被Loop化
2. Loop化解放"操作型工作"，聚焦"设计型工作"
3. 好的Loop = 好的停止条件 + 好的验证机制
4. 你不是在写流程，你是在设计"能写流程的系统"
```

## 🍅 番茄92 · T2：Loop六块积木——Automation/Worktree/Skill/Connector/Sub-agent/Memory

### 🔍 犯罪现场：搭一个Loop就像搭乐高

一个完整的Loop不是大杂烩。它由六个标准化的组件组成——Addy Osmani提出的"Loop六块积木"。

### 🧠 六块积木概览

```
┌─────────────────────────────────────────────────────────────┐
│                    一个完整的Loop系统                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│  │Automation   │    │  Worktree    │    │   Skill      │   │
│  │定时/事件触发 │    │  隔离工作环境  │    │  Agent入职手册│   │
│  └─────────────┘    └──────────────┘    └──────────────┘   │
│                                                             │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│  │Connector    │    │  Sub-agent   │    │   Memory     │   │
│  │外部工具连接  │    │  分工与制衡   │    │   持久化状态  │   │
│  └─────────────┘    └──────────────┘    └──────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 🧠 积木一：Automation（自动化触发器）

**作用**：决定Loop什么时候启动。

| 触发类型 | 例子 | 工具 |
|:---------|:-----|:-----|
| **定时触发** | 每天早上8点检查部署状态 | cron, GitHub Actions Schedule |
| **事件触发** | 有新Issue时自动分析 | GitHub Webhook, Slack Event |
| **条件触发** | 当错误率超过5%时启动诊断 | Prometheus Alert, Datadog Monitor |
| **手动触发** | 用户输入`/goal`命令 | Claude Code, Codex CLI |

**提问**：没有自动化的Loop不是Loop，只是一个手动流程。你的Loop第一步是什么？

### 🧠 积木二：Worktree（工作树隔离）

**作用**：让多个Agent工作在隔离环境中，互不干扰。

```
主分支 main
  ├── worktree/agent-1  ← Agent 1 在这里写代码
  ├── worktree/agent-2  ← Agent 2 在这里审查代码
  └── worktree/agent-3  ← Agent 3 在这里写测试
```

**为什么需要？** 如果两个Agent同时修改同一个文件，会发生冲突。Worktree给每个Agent一个"独立副本"。

### 🧠 积木三：Skill（技能知识库）

**作用**：给Agent一份"入职手册"，减少猜测成本。

不加Skill的Loop：Agent每次都要猜测"这个项目的测试风格是什么？"
加了Skill的Loop：Agent第一次就读到规则，不再猜测。

（T3会专门讲Skill Engineering，这里不展开）

### 🧠 积木四：Connector（插件/连接器）

**作用**：让Agent能操作真实世界的工具。

| 连接器 | 让Agent能做什么 |
|:-------|:--------------|
| GitHub Connector | 读Issue、提PR、Review代码 |
| Filesystem Connector | 读写文件、创建目录 |
| Slack Connector | 发消息、读频道 |
| Database Connector | 查询、修改数据库 |
| Browser Connector | 浏览网页、填表单 |
| Email Connector | 收发邮件 |

**Connector的本质**：把Agent的"语言能力"翻译成"工具操作能力"。

### 🧠 积木五：Sub-agent（子代理分工）

**作用**：专业化分工 + 制衡机制。

```
最核心的模式——Maker-Checker：
  
  Maker：负责"生产"（写代码、写文章、生成方案）
    关注点：速度、产出量
  
  Checker：负责"验证"（审代码、审事实、审安全）
    关注点：质量、正确性、安全性
  
  ⚠️ 关键原则：Maker和Checker不能是同一个Agent
```

### 🧠 积木六：Memory/State（持久化状态）

**作用**：让Loop有"记忆力"——每次迭代记得之前做了什么。

```
状态文件 (.loop-state/task-42.md)：
  ## 当前状态
  - 分析阶段：✅ 完成
  - 实现阶段：🔄 进行中（60%）
  - 测试阶段：⏳ 等待
  
  ## 已尝试的方案
  1. 方案A：因性能问题被否决
  2. 方案B：实现中，预计可行
  
  ## 下次迭代注意事项
  - 检查边界情况（空列表）
```

> ✋ **费曼自测**：六块积木中，你认为哪一块"最容易被忽视但最关键"？为什么？

### 🧠 六块积木协作流程

```
事件触发（Automation）
    ↓
创建工作树（Worktree）
    ↓
加载技能（Skill） + 连接工具（Connector） + 读取状态（Memory）
    ↓
启动子代理循环（Sub-agent）：
  Maker执行 → Checker验证
    ↓
  不通过 → 反馈Maker重新执行
  通过 → 更新状态，合并，发送通知
    ↓
Loop结束或等待下一次触发
```

## 🍅 番茄93 · T3：Skill Engineering——SKILL.md：给Agent的"入职手册"

### 🔍 犯罪现场：为什么同样一个Agent，在你的项目里是天才，在别人的项目里是白痴？

差距通常在**Skill**——你有没有给Agent一份好的"入职手册"。

### 🧠 SKILL.md是什么

**SKILL.md = 给Agent看的"入职手册"，不是给人看的文档。**

```
传统文档（README.md）：
  写给"人类开发者"看 → 讲究可读性、排版、上下文

Skill文件（SKILL.md）：
  写给"AI Agent"看 → 讲究指令清晰、规则明确、示例丰富
```

### 🧠 好的SKILL.md包含什么

```
┌────────────────────────────────────────┐
│  SKILL.md 标准结构                      │
├────────────────────────────────────────┤
│                                        │
│  1. 技能名称和一句话描述（"这个技能是做什么的"）│
│                                        │
│  2. 适用范围（"什么条件下应该/不应该调用"）   │
│                                        │
│  3. 项目规则（代码风格、命名规范、测试要求）   │
│                                        │
│  4. 最佳实践（"记住要…"、"千万不要…"）     │
│                                        │
│  5. 示例（3-5个最常见的场景 + Agent应该怎么做）│
│                                        │
│  6. 常见陷阱（"特别要注意…"）             │
│                                        │
│  7. 检查清单（完成后的自检项目）             │
│                                        │
└────────────────────────────────────────┘
```

### 🧠 SKILL.md vs CLAUDE.md

在你的项目环境中：

| 文件 | 作用范围 | 内容 |
|:-----|:---------|:-----|
| **CLAUDE.md** | 全局项目（在根目录） | 项目结构、代码风格、通用规则 |
| **SKILL.md** | 特定技能（在.skills/或命令调用） | 该技能的任务流程、规则、示例 |
| **MEMORY.md** | 知识库（vault级别） | 知识库结构、目录说明、状态文件 |

### 🧠 写SKILL.md的核心原则

#### 原则一：指令要"可验证"，不要"喊口号"

```
❌ 差："保证代码质量"
✅ 好："所有函数必须有类型注解。所有公共方法必须有docstring。
        测试覆盖率必须 ≥ 80%。每次提交前运行 pytest --cov。"
```

#### 原则二：给示例，不要只给定义

```
❌ 差："使用一致的命名风格"
✅ 好：
  ✅ 正确示例：
    def get_user_by_id(user_id: int) -> User:
    class UserService:
    MAX_RETRY_COUNT = 3
    
  ❌ 错误示例：
    def GetUserByID(uid):
    class user_service:
    maxRetry = 3
```

#### 原则三：说"什么条件下启动"，不要只说"做什么"

```
❌ 差："这个技能用来写测试"
✅ 好："当用户说'写测试'或'增加测试覆盖'或提交PR包含新功能时，
        调用此技能。不要在以下情况下调用：① 用户明确说不需要测试
        ② 只是修改了文档 ③ 纯配置变更。"
```

### 🧠 一个SKILL.md示例片段

```markdown
# Skill: Python代码审查

## 一句话描述
用严格的标准审查Python代码，关注正确性、性能、可维护性、安全性。

## 适用范围
- ✅ PR中有Python文件变更时
- ✅ 用户主动要求代码审查时
- ❌ JavaScript/C++代码（这是另一个技能）
- ❌ 文档/配置文件的简单变更

## 审查规则（按优先级）

### P0 - 必须检查
- [ ] 所有函数有类型注解（Type Hints）
- [ ] 异常处理——没有裸的 `except:`
- [ ] SQL注入风险——使用参数化查询
- [ ] 路径遍历风险——验证文件路径

### P1 - 建议检查
- [ ] 函数长度不超过50行
- [ ] 没有重复代码（DRY原则）
- [ ] 测试覆盖新代码 ≥ 80%

## 示例

PR添加了以下功能：
```python
def get_user(uid):
    return db.query(f"SELECT * FROM users WHERE id = {uid}")
```

审查意见：
🔴 P0漏洞：SQL注入风险。应使用参数化查询：
```python
def get_user(uid: int) -> dict | None:
    return db.query("SELECT * FROM users WHERE id = ?", (uid,))
```
```

> ✋ **费曼自测**：写好SKILL.md和"带实习生"有什么共同点？为什么"给示例"比"说要求"更有效？

### 🧠 CLAUDE.md实战写法（你的项目示例）

参考本项目根目录的CLAUDE.md，你会发现它其实也是一个"给Agent的入职手册"：

```markdown
# CLAUDE.md - 关键要素

## 角色定位
明确告诉Agent：你是谁、你叫什么、你的职责是什么。

## 行为准则
告诉Agent：应该用什么风格、怎么思考、怎么操作。

## 路径规范
告诉Agent：文件路径规则——Vault文件用什么路径，导出文件用什么路径。

## 三层记忆系统
告诉Agent：启动时先读什么，工作时怎么保存，结束时怎么总结。
```

**你的CLAUDE.md本质上就是一个Skill文件——它定义了你希望Agent如何在这个知识库中工作。**

## 🍅 番茄94 · T4：Loop Engineering实战——用Claude Code /goal和/loop设计自运行系统

### 🔍 犯罪现场：把你的Skill变成"自己跑起来的系统"

纸上谈兵到此为止。

现在，你将学习如何用实际工具（Claude Code）把你的Skill变成一个自运行系统。

### 🧠 两个核心命令

| 命令 | 功能 | 类比 |
|:-----|:-----|:-----|
| **`/loop`** | 定时循环——每隔N分钟执行一次指定prompt | "每5分钟检查一次邮箱" |
| **`/goal`** | 目标驱动循环——持续执行直到满足条件 | "把房间打扫干净，打扫完停" |

### 🧠 `/loop` 实战

```
/loop 5m "检查生产环境部署状态，如果有异常立即报告"

Agent会怎么做：
  每5分钟执行一次 →
    Thought: 检查部署状态
    Action: 调用监控API获取数据
    Observation: 正常/异常
    Thought: 如果有异常 → 发送告警
            如果正常 → 记录日志，等待下一轮
  直到：用户停止 / 达到最大次数
```

**/loop的典型场景**：
- 定时检查部署状态
- 监控关键指标（错误率、响应时间）
- 定期抓取网页内容
- 轮询等待某个条件（如"等CI通过"）

### 🧠 `/goal` 实战

```
/goal "修复项目中的所有lint错误"

Agent会怎么做：
  第1轮 →
    Thought: 需要找出所有lint错误
    Action: 运行 lint 工具
    Observation: 发现15个错误
    Thought: 开始修复第一个
    Action: 修复...
    
  第2轮 →
    Action: 再次运行 lint 检查
    Observation: 还有3个错误
    Thought: 继续修复
    
  第3轮 →
    Action: 再次运行 lint 检查
    Observation: 0个错误 ✅
    Thought: 目标完成
    Action: 输出报告
```

**/goal的典型场景**：
- "修复所有bug"
- "完成这个功能的所有测试"
- "把所有文档从旧格式迁移到新格式"
- "优化网站性能指标到目标值"

### 🧠 /loop vs /goal 对比

| 维度 | /loop | /goal |
|:-----|:------|:------|
| **停止条件** | 时间或次数 | 目标达成 |
| **节奏** | 固定间隔执行 | 连续执行直到完成 |
| **适用场景** | 监控、轮询、定期任务 | 批量处理、代码修复、迁移 |
| **风险** | 可能空跑（无异常时也在检查） | 可能死循环（目标达不到时） |

### 🧠 设计你的第一个Loop流程

```
               ┌─────────────────────────────────┐
               │    Step 1：定义目标               │
               │    "修复Issue #42中的bug"         │
               └──────────────┬──────────────────┘
                              │
               ┌──────────────▼──────────────────┐
               │    Step 2：定义停止条件            │
               │    □ 所有测试通过                  │
               │    □ lint检查通过                  │
               │    □ 代码审查通过                   │
               │    □ PR已创建                      │
               └──────────────┬──────────────────┘
                              │
               ┌──────────────▼──────────────────┐
               │    Step 3：配置六块积木            │
               │    触发 → 隔离 → 加载技能          │
               │    → 连接工具 → 启动子Agent       │
               │    → 持久化状态                   │
               └──────────────┬──────────────────┘
                              │
               ┌──────────────▼──────────────────┐
               │    Step 4：启动并监控              │
               │    "开始执行" → 观察第一轮          │
               │    → 检查结果 → 优化配置           │
               └─────────────────────────────────┘
```

### 🧠 实际场景：用Loop自动修复一个Bug

```
你的输入：
  /goal "分析并修复src/buggy_module.py中导致测试失败的问题"

背后的Loop：
  
  Round 1：分析
    → 读取测试失败的log
    → 分析buggy_module.py源码
    → 定位：第42行忘记处理None的情况
    → 状态更新："已定位根因"
  
  Round 2：修复
    → 加None检查 + 测试用例
    → 运行测试 → 通过 ✅
    → 但lint发现有格式问题
  
  Round 3：修复lint
    → autoformat代码
    → 运行lint → 通过 ✅
    → 运行测试 → 仍然通过 ✅
    → 状态更新："修复完成，所有检查通过"
  
  Round 4：提PR
    → 创建PR，附上修复说明
    → 目标达成 → Loop结束
```

> ✋ **费曼自测**：/goal和/loop的核心区别是什么？什么时候该用哪个？用"打扫房间"做类比。

### 🧠 实践经验：Loop设计的黄金法则

| 法则 | 说明 | 反例 |
|:-----|:-----|:-----|
| **① 先定义停止条件** | 在启动前就想清楚"什么时候算完成" | "帮我优化一下" → 永远优化不完 |
| **② 设最大迭代次数** | 防止死循环烧Token | 不设上限 → 跑了100轮还在跑 |
| **③ 状态外部化** | 每轮结束后写入状态文件 | 用Agent记忆 → 重启后忘记进度 |
| **④ Maker-Checker分离** | 生产者和验证者不同 | 同一个Agent写+审 → 漏掉低级错误 |
| **⑤ 人留否决权** | 重要决策前通知人类 | 全自动 → 提交了包含bug的PR |

### ✅ 费曼三句话——Part 4总复习

1. **大语言模型是什么？** ——一个通过"猜下一个词"训练出来的超大规模系统，在猜词过程中涌现了语法、知识、推理和创造力；它的能力受限于训练数据截止时间和Context窗口大小。
2. **如何让LLM更"有用"？** ——通过Prompt Engineering（教它怎么回答）+ Context Engineering/RAG（给它需要的知识）+ Agent架构（给它行动能力和记忆）——这三层叠加，让LLM从"聊天机器人"变成"智能系统"。
3. **Loop Engineering解决了什么根本问题？** ——把人从"AI操作员"角色解放出来，变成"AI系统架构师"；你不是在写每一条Prompt，而是在设计一个能自我运行、自我迭代的AI系统。
